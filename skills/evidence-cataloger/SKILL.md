# Evidence Cataloger

## When to invoke

- `storage-incident-response` calls this as Phase 4 (after evidence-collection-orchestrator)
- Each round produces its own catalog + manifest sealed with the round's SHA-256 set
- Standalone: when sealing any forensic evidence bundle for later attestation

## The principle

```
EVIDENCE WITHOUT PROVENANCE IS A RUMOR
```

Per NIST SP 800-86 (Forensic Techniques for Incident Response) and ISO/IEC 27037 (Digital evidence handling), every artifact must have:

1. **Integrity hash** — SHA-256 (MD5 and SHA-1 are deprecated for forensic use)
2. **Chain of custody record** — who collected it, when, what command, source host
3. **Tamper evidence** — append-only custody.log, externally witnessed bundle hash

Without these, an analyst cannot defend the report against "your evidence was modified" claims.

## Bundle hashing

```bash
ROUND_DIR="<handoff>/<id>/round-<N>"
cd "$ROUND_DIR"

# 1. Compute per-artifact hashes
find evidence/ -type f \( ! -name manifest.sha256 ! -name custody.log ! -name catalog.md \) -print0 \
  | xargs -0 sha256sum > manifest.sha256

# 2. Compute bundle hash (deterministic — sort manifest first, then hash)
sort manifest.sha256 | sha256sum | awk '{print $1}' > bundle.sha256.txt

# 3. (Optional) External witness — commit bundle hash to separate audit git repo
# (Optional) RFC 3161 timestamp the bundle hash via openssl ts
```

## Hard rules

- **SHA-256 only.** MD5 and SHA-1 are forbidden for forensic use (NIST SP 800-86 §6.4).
- **Append-only `custody.log`.** Never edit prior lines. If correcting, append a new line stating the correction.
- **Sort manifest.sha256 before computing bundle hash** — bundle hash must be reproducible.
- **One catalog per round.** Cross-round summary lives at `<handoff>/<id>/manifest.sha256` (merge of all round-N manifests) but each round's own manifest stays untouched.
- **Never modify evidence files.** If a file is too large to ship, gzip the ORIGINAL during collection (recompute sha256 after gzip); never recompress after the manifest is sealed.
- **Document gaps explicitly.** `UNREACHABLE` and `TIMEOUT` and `PERMISSION_DENIED` are valid custody.log entries — silence is not.

## Standalone usage (non-incident)

For ad-hoc forensic sealing (e.g., snapshot of a workload before destructive op):

```bash
EVIDENCE_DIR=<dir>
cd "$EVIDENCE_DIR"
find . -type f ! -name manifest.sha256 -print0 | xargs -0 sha256sum > manifest.sha256
sort manifest.sha256 | sha256sum
```
