# S1 Verdict — Gatekeeper & Policy Spine

**Question.** Can Cedar (in-process) + an embedded gateway + a chained audit
ledger meet the governed-mutation path budgets, and should we embed an
agentgateway-class OSS gateway or build a thin custom MCP-aware proxy?

**Overall: GO.** Cedar (WASM, preparsed + entity-sliced) meets NF-004 with two
orders of magnitude of headroom on the per-call path; the chained ledger meets
NF-003 with three orders of magnitude of headroom; payload-hash invalidation
blocks tampering deterministically; the offline verifier rejects any tamper and
re-derives the Merkle root. **Embed agentgateway** (Apache-2.0, MIT-compatible,
MCP-native, has the exact extension points we need) rather than building a proxy.

All numbers measured locally on `node v22.12.0, linux/x64` against a
programmatically generated reference set of **208 policies / 6002 entities**
(100 teams, 400 agents, 5500 tools). Reproduce with `npm run bench`.

---

## EC1 — Cedar P99 ≤ 100 ms at reference scale (≥200 policies, ≥5000 entities)

**Verdict: PASS** (per-call); **PASS, marginal** (tool-listing). Both phases
evaluated.

| Phase | Strategy | P50 | P95 | P99 | Budget | Result |
|---|---|---|---|---|---|---|
| (b) invocation (per-call) | preparsed pset + sliced entities | 0.435 ms | 0.633 ms | **0.764 ms** | 100 ms | PASS |
| (a) tool-list (200 tools/listing) | per-tool stateful, single-tool slice | 87.5 ms | 90.0 ms | **91.1 ms** | 100 ms | PASS |
| (a) per-tool amortised | — | — | — | ≈0.44 ms | — | — |

**Worst-case anti-patterns (why the design above is mandatory):**

| Anti-pattern | P50 | P99 | Result |
|---|---|---|---|
| naive `isAuthorized` (re-parse pset + entities every call) | 128 ms | **140.6 ms** | FAIL |
| `statefulIsAuthorized` + FULL 6k-entity store per call | 116 ms | **128.1 ms** | FAIL |

**Methodology.** `@cedar-policy/cedar-wasm@4.11.1` (official AWS Cedar WASM
binding, Apache-2.0), `nodejs` synchronous build. Reference set generated by
`src/reference-set.ts`. Invocation: 1000 warm + 1000 timed single calls across a
deterministic agent×tool mix; `performance.now()` per call; percentiles by sort.
Tool-listing: 200-tool catalogue, 5 warm + 20 timed listings, per-listing wall
time. Phase (a) uses partial evaluation (`isAuthorizedPartial`) only when one
residual collapses to a uniform decision; otherwise it concretises per tool with
`statefulIsAuthorized`.

**The load-bearing finding.** The WASM binding re-parses (1) the policy set and
(2) the entity store **on every `isAuthorized` call**. At reference scale that is
~140 ms P99 — over budget. The mitigations, both implemented in
`src/cedar-engine.ts`, are required for production:

1. **Preparse the policy set once** — `preparsePolicySet(id, pset)` at engine
   construction, then `statefulIsAuthorized({ preparsedPolicySetId, ... })` on the
   hot path. Eliminates policy re-parse.
2. **Pass the minimal entity slice** — principal + its group chain + the resource
   + its group chain + the action (~3–5 entities), not the whole store. Per-call
   entity-parse cost scales with slice size: full store ≈18 ms, single-tool slice
   ≈0.5 ms.

**Caveat for tool-listing.** `isAuthorizedPartial` has **no** stateful (preparsed)
variant in the WASM binding, so a residual pass carries the fixed ~140 ms
re-parse cost; it is therefore *not* the primitive for large catalogues. The
spike filters via N cheap stateful per-tool calls (O(N), ~0.44 ms each). A 200-
tool listing lands at 91 ms P99 — inside budget but with little headroom; for
larger catalogues see "Suggested amendments" below (cache by policy-version +
agent-scope; the MCP tools/list result is highly cacheable).

## EC2 — Ledger write on the mutation path ≤ 25 ms P99; checkpoint cadence + proof size

**Verdict: PASS.**

| Metric | P50 | P95 | P99 | Budget | Result |
|---|---|---|---|---|---|
| single AuditRecord append (canonical-JSON hash + chain link) | 0.006 ms | 0.011 ms | **0.016 ms** | 25 ms | PASS |
| 4-record mutation-path hashing (decision, dry-run, approval, execute) | — | — | **≈0.066 ms** | 25 ms | PASS |

The measured figure is **hashing + chain-link CPU only**; durable Postgres I/O is
out of scope for this spike (no DB credentials — `ENVIRONMENT REALITY`). The
budget headroom (25 ms vs. 0.066 ms) leaves ~24.9 ms for a single synchronous
`INSERT ... RETURNING` on the append-only table, which is comfortable for a
local/region Postgres. **The hash chain is not the bottleneck; the DB write is.**
Recommendation: keep the chain hashing on the request path (it is free) and use a
single-row insert per record; batch the Merkle checkpoint off the request path.

**Merkle checkpoint cadence vs. proof size** (measured over a 22k-record ledger):

| cadence (records) | tree depth | inclusion-proof size | checkpoint build P99 |
|---|---|---|---|
| 256 | 8 | 256 bytes | 4.1 ms |
| 1024 | 10 | 320 bytes | 7.2 ms |
| 4096 | 12 | 384 bytes | 13.5 ms |
| 16384 | 14 | 448 bytes | 57.8 ms |
| 22000 | 15 | 480 bytes | 77.2 ms |

Proof size grows as `32 · ceil(log2(N))` bytes (one 32-byte sibling digest per
tree level; domain-separated leaf `0x00` / node `0x01` hashing per RFC 6962
spirit). **Recommended cadence: 1024 records per checkpoint** — depth 10, a
320-byte inclusion proof, and a 7 ms build cost that runs **off** the mutation
path (a background worker exports the root to customer object storage, per spec
§3). Cadence is a tunable; even at 22k records the proof is only 480 bytes, so
cadence can be relaxed if checkpoint frequency is the binding constraint.

Checkpoint building is never on the mutation path, so it does not count against
the 25 ms budget; it is reported here to satisfy the "proof-size data" criterion.

## EC3 — agentgateway embed verdict (license + extension points)

**Verdict: PASS — EMBED agentgateway.** Build a thin control-plane shim, not a
gateway.

**License: compatible.** agentgateway is **Apache-2.0** (LICENSE: "Copyright 2017
Solo.io, Inc.", Apache License 2.0). Apache-2.0 is compatible with our MIT
platform — Apache-2.0 components may be distributed within an MIT-licensed
product provided the Apache NOTICE/attribution is preserved. No copyleft.
Governance: donated by Solo.io, now an **Agentic AI Foundation (AAIF) project
under the Linux Foundation** — durable, vendor-neutral stewardship.

**Architecture.** Predominantly **Rust** (~63%) with Go and TS components; a
single data plane for HTTP/gRPC plus AI-native protocols. **MCP is first-class**:
it treats MCP servers like microservices with discovery, versioning, RBAC, and
audit trails, and supports per-tool MCP authorization on verified JWT claims.

**Extension points sufficient for PolicyDecisionRecord injection + signed
decision logging — yes:**

- **External authorization (`ext_authz`)** — "bring your own external
  authorization service to protect requests" (both standalone and Kubernetes
  modes). This is the seam for our Cedar PDP: the gateway calls out to our Policy
  Gateway (C1), which returns permit/deny **and** emits the normalized
  `PolicyDecisionRecord` into our ledger. The gateway enforces; we decide and
  record. This matches our trust-boundary model (decision + audit stay in our
  control plane).
- **External processing (ExtProc)** — modify request/response via an external
  server. Lets us inject the dry-run gate, the ApprovalObject hand-off, and the
  payload-hash re-validation on the execute hop without forking the gateway.
- **CEL RBAC + OTel logs/metrics/traces** — native fine-grained RBAC and
  tamper-evident OTel audit hooks for defense-in-depth beneath our Cedar layer.
- **Custom webhooks / content filtering** — additional inline control points.

**Why not build a proxy.** A custom MCP-aware proxy would re-implement: MCP
session multiplexing, JWT/OAuth/API-key auth, RBAC, rate limiting, TLS/mTLS,
content filtering, OTel wiring, and per-tool MCP authz — all of which
agentgateway already ships and the platform research explicitly says "do not
rebuild the gateway layer." Estimated avoided cost: **~2–3 engineer-months** for
a credible v1 plus ongoing maintenance of a security-critical network component.

**Fallback (documented, not recommended).** If a future agentgateway change
removes `ext_authz`/ExtProc or its MCP authz proves too coarse, the fallback is a
thin Rust/TS MCP proxy that (a) terminates MCP, (b) calls our PDP over the same
`ext_authz` contract, (c) forces dry-run + ApprovalObject, (d) re-validates the
payload hash, (e) writes the chained ledger. Cost estimate: **~2–3 eng-months**
to first usable cut, then sustained ownership. Because we keep the PDP, approval,
and ledger in our control plane regardless, the gateway is replaceable behind a
stable internal interface — so embedding now is low-risk.

## EC4 — Payload-hash invalidation blocks a mutated payload (GOV-004 invariant)

**Verdict: PASS.** Demonstrated in `npm run flow` step [3] and enforced by
`test/payload-hash-invalidation.test.ts`.

The `ApprovalObject.payload_hash` pins the canonical-JSON sha256 of the exact
execution payload at approval time. On the execute hop the gatekeeper recomputes
the hash of the payload it is about to run and compares:

- **Unchanged payload** → hashes match → executes (`state: executed`). The mock
  target is invoked exactly once; the executed payload's hash equals the approved
  hash.
- **Mutated payload** (e.g. `replicas: 6` → `600` after approval) → hashes differ
  → **BLOCKED**: `state: invalidated`, a first-class **denial** AuditRecord is
  written (`detail_ref: payload_hash_mismatch`), and **the mock target is never
  called** (nothing reaches the simulated cluster). Test asserts
  `executor.applied.length === 0`.
- **Key reorder only** (same values, different JSON key order) → canonicalization
  sorts keys → hashes match → executes. No false positive.
- **Rejected approval** → never executes.

The approver's decision log entry records `payload_hash_seen` equal to the
approved `payload_hash` (decision-log integrity).

## Suggested amendments to spec Part 0 §3 / Part 1

1. **§3 "Policy engine" — make the hot-path technique normative.** Add: the Cedar
   `PolicyEngine` MUST preparse the policy set per tenant (cache by
   tenant + policy-set version) and MUST pass only the request-relevant entity
   slice; calling `isAuthorized` with the full policy + entity set per request is
   over the NF-004 budget at reference scale (measured ~140 ms P99 vs. 0.76 ms
   sliced). This is an implementation requirement, not just a benchmark note.

2. **§3 — record the partial-evaluation limitation.** `isAuthorizedPartial` in
   the current WASM binding has no preparsed/stateful variant and re-parses the
   policy set (~140 ms at reference scale). Tool-list filtering should therefore
   use cached per-tool stateful evaluation, and the MCP `tools/list` result
   SHOULD be cached per (agent-scope, policy-set version) — it changes only when
   policy or scope changes. Spec should name this cache as a component concern of
   C1. Consider tracking the native Cedar (Rust) crate behind the same
   `PolicyEngine` interface for the gateway-side enforcement path, where a
   preparsed authorizer object avoids both re-parse costs entirely.

3. **§3 "Audit ledger" — confirm cadence + clarify the budget.** Adopt **1024
   records/checkpoint** (depth 10, 320-byte proof) as the default cadence. Clarify
   that the NF-003 ≤25 ms budget is dominated by the durable Postgres insert, not
   the hash chain (chain hashing is ~0.016 ms P99); the budget should be stated
   against the row insert, and checkpoint building MUST stay off the mutation
   path.

4. **§3 "Tool-call gateway" — resolve in favor of EMBED agentgateway**
   (Apache-2.0, AAIF/LF, MCP-native, `ext_authz`+ExtProc). The custom-proxy line
   can be downgraded from an alternative to a documented fallback with the
   ~2–3 eng-month estimate above; keep PDP/approval/ledger in our control plane so
   the gateway stays replaceable behind a stable internal interface.

5. **Part 1 — schemas validated as-is; no changes required.** The prototype emits
   `PolicyDecisionRecord`, `ApprovalObject`, and `AuditRecord` that pass the real
   Ajv validators unchanged. One ergonomics note (non-blocking): `AuditRecord.hash`
   is defined as `sha256(canonical(record without hash) || prev_hash)`; the spike
   confirms this is well-defined and verifiable offline, but Part 1 could state
   the canonicalization rule (recursive key sort) explicitly so independent
   verifiers reproduce the digest byte-for-byte.
