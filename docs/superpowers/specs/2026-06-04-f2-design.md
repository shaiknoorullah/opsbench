## 1. Purpose

opsbench's evidence guarantee depends on a single, auditable choke-point through which every agent → MCP tool call must flow. Today every recipe wires the agent directly to the upstream MCP server, which means Cedar evaluation happens inside the agent process (Pi, Claude Code, Codex CLI, …) and the `custody.log` is whatever each host happens to write. F2 fixes that by shipping `opsbench-gateway`: a fork of `stacklok/toolhive` that adds Cedar policy evaluation, redacted-args custody-log emission, and a uniform gateway URL the recipes target. Pi and the other coding hosts continue to think they are talking to N upstream MCPs; the gateway demultiplexes, evaluates, redacts, logs, and forwards. This phase ships *only* the gateway plus the recipe rewrites needed to point at it — Ed25519 signing of the custody log lives in F3, multi-host adapters live in F5, team-package wiring lives in F4.

## 2. Scope

### In-scope

- **Fork base.** Vendor `stacklok/toolhive` (Apache-2.0, commit pinned in `packages/opsbench-gateway/UPSTREAM`) into `packages/opsbench-gateway/`. Apache-2.0 is compatible with opsbench's MIT root; vendored source carries the upstream `LICENSE` and a top-level `NOTICE` file enumerating the toolhive copyright + opsbench changes.
- **New Go package.** `packages/opsbench-gateway/internal/cedar/` — Cedar evaluator that wraps `github.com/cedar-policy/cedar-go` (the official Go implementation) and merges the two F1 policy files at startup.
- **New Go package.** `packages/opsbench-gateway/internal/custody/` — JSON-Lines custody log writer with redaction, rotation, and a stable schema (`schemas/custody-log.v1.json`).
- **Config file.** `gateway.yaml` (loaded from `$OPSBENCH_GATEWAY_CONFIG` or `~/.config/opsbench/gateway.yaml`) declares upstream MCPs, policy paths, custody-log path, listen address, redaction rules.
- **Distribution.** Two artifacts per release:
  1. Docker image `ghcr.io/shaiknoorullah/opsbench-gateway:<semver>` (multi-arch: linux/amd64, linux/arm64, darwin/arm64).
  2. Statically-linked binary `opsbench-gateway-<os>-<arch>` attached to the GitHub Release (linux/macOS/x86_64 + arm64).
- **CLI surface.** Cobra-based subcommands inherited from toolhive plus opsbench additions: `serve`, `policy lint`, `policy explain`, `custody verify-format`, `version`. Pi calls `serve` from a long-running process supervisor (systemd/launchd snippets shipped in `packages/opsbench-gateway/dist/`).
- **Recipe rewrites.** Every F0 recipe gains a "Configuration — opsbench-gateway (recommended)" section that becomes the lead config block. The original direct-connection block is preserved under "Configuration — direct (advanced)".
- **Pi-first config docs.** Pi's `AGENTS.md` snippets in each recipe point at `http://localhost:8765/mcp/<upstream-id>` (the gateway's per-upstream sub-path), not the upstream server's stdio command.
- **CI.** `.github/workflows/gateway-build.yml` builds the binary + image on every PR that touches `packages/opsbench-gateway/**`; `.github/workflows/gateway-integration.yml` runs a Docker-Compose integration test with three upstream MCPs (a fake stdio MCP, `containers/kubernetes-mcp-server`, `pab1it0/prometheus-mcp-server`).
- **Upstream-sync cadence doc.** `packages/opsbench-gateway/UPSTREAM-SYNC.md` documents the monthly merge process from `stacklok/toolhive` `main`.

### Out-of-scope

- **Ed25519-signed receipts.** F3 owns the signing schema, key bootstrap, verifier CLI, and the schema-version bump. F2 emits SHA-256 hashes of args/responses; the field names align with what F3 will populate so the schema bump in F3 is purely additive.
- **Distributed deployment.** Single-host only. No clustering, no horizontal scale, no shared state. Multi-instance is a F6 evaluation.
- **Multi-tenancy.** One gateway instance per user/agent. The `principal` in Cedar is read from `gateway.yaml`'s `agent_class` field; the gateway does not authenticate distinct end-users.
- **Web UI / Grafana dashboard.** Gateway runs headless. Observability is custody.log + the Grafana MCP recipe pointed at the gateway's own `/metrics` Prometheus endpoint.
- **MCP protocol extensions.** F2 implements MCP `stdio` + `streamable HTTP` transports as toolhive already does. WebSocket / SSE additions are deferred until upstream toolhive adopts them.
- **agentgateway integration.** F6 evaluates whether opsbench-gateway folds into agentgateway or stays standalone. F2 ships standalone.
- **Backfilling existing direct-connection users.** Recipes document both paths; the gateway is recommended, the direct path is a documented fallback. Users on direct paths continue to work; no breaking change.
- **Per-tool rate limiting.** F2 has a global rate limit (`gateway.yaml#rate_limit`). Per-tool / per-agent rate limits are a F4 / F6 concern.

### 3.2 Per-request lifecycle

```text
1. agent opens transport to :8765/mcp/<upstream-id>
2. gateway accepts connection, looks up <upstream-id> in gateway.yaml#upstreams
3. agent sends MCP request (tool/call, resources/read, prompts/get, …)
4. router parses MCP envelope, extracts:
       agent_class  ← gateway.yaml#agent_class
       action       ← "<upstream-id>::<tool-name>"   (e.g. "vault-mcp::vault::kv::read")
       resource     ← derived from request args via per-upstream resource_shape map
       context      ← {human_approval, incident_round, incident_phase, …}
                       merged from request headers (X-Opsbench-Context-*) and gateway.yaml defaults
5. cedar evaluator runs against tools.cedar + tools-generated.cedar; tools.cedar wins on conflict
       decision ∈ {allow, deny}
6. if deny:
       gateway returns MCP error -32603 (Internal error) with structured `data` field:
           {decision: "deny", reason: "<cedar reason-id or default-deny>", policy_file: "<path>"}
       custody.log line written with decision=deny, response_sha256=null
7. if allow:
       gateway forwards request to upstream MCP child process (stdio) or HTTP endpoint
       awaits response (with timeout per gateway.yaml#upstream_timeout)
       writes custody.log line:
           ts, gateway_id, agent_class, upstream_id, tool, args_sha256, response_sha256,
           decision="allow", latency_ms, redacted_args, redacted_response_preview
       returns upstream response to agent
8. on any panic / upstream error, gateway writes custody.log line with decision="allow",
       error_class, and a non-zero exit_code so downstream tooling can replay
```

### 3.3 Wire-format custody-log entry (F2; F3 extends)

`schemas/custody-log.v1.json` (lives at `packages/opsbench-gateway/schemas/custody-log.v1.json`):

```jsonc
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://opsbench.dev/schemas/custody-log.v1.json",
  "type": "object",
  "required": [
    "schema_version", "ts", "gateway_id", "agent_class",
    "upstream_id", "tool", "decision", "args_sha256"
  ],
  "properties": {
    "schema_version": { "const": "v1" },
    "ts":             { "type": "string", "format": "date-time" },
    "gateway_id":     { "type": "string", "description": "UUIDv4 generated once per gateway process; written to ~/.local/state/opsbench/gateway-id" },
    "agent_class":    { "type": "string" },
    "upstream_id":    { "type": "string", "description": "key from gateway.yaml#upstreams" },
    "tool":           { "type": "string" },
    "decision":       { "enum": ["allow", "deny", "error"] },
    "deny_reason":    { "type": "string" },
    "args_sha256":    { "type": "string", "pattern": "^[a-f0-9]{64}$" },
    "response_sha256":{ "type": ["string", "null"], "pattern": "^[a-f0-9]{64}$" },
    "latency_ms":     { "type": "integer", "minimum": 0 },
    "redacted_args":  { "type": "object", "additionalProperties": true },
    "redacted_response_preview": { "type": ["string", "null"], "maxLength": 4096 },
    "policy_files":   { "type": "array", "items": { "type": "string" } },
    "context":        { "type": "object", "additionalProperties": true },
    "error_class":    { "type": ["string", "null"] }
  },
  "additionalProperties": false
}
```

F3 will introduce `custody-log.v2.json` that adds `signer_pubkey` and `signature` while keeping v1 fields intact. Gateway will read either version; writer is configurable.

### 3.4 Redaction pipeline

Redaction runs before any args hit the disk or the cedar evaluator's `context`. Two-layer:

1. **Structural redaction** declared per upstream in `gateway.yaml#upstreams[].redaction`:
   - `redact_keys`: list of JSON-pointer paths (e.g. `/args/password`, `/args/data/token`).
   - `redact_patterns`: list of named regex patterns from the global `gateway.yaml#redaction_patterns` table (e.g. `aws_access_key_id`, `bearer_token`, `pem_block`).
2. **Default-deny redaction.** If `gateway.yaml#redaction_mode: strict`, any key whose name matches a deny-list of common secret words (`password`, `secret`, `token`, `apikey`, `private_key`, …) gets replaced with `"[REDACTED:keyname]"` regardless of upstream config. Strict is the default in shipped configs; `permissive` is opt-in.

Redacted args are what get written to `custody.log#redacted_args`. The raw args are SHA-256-hashed (after canonicalisation: keys sorted, whitespace stripped) and that hash goes in `custody.log#args_sha256`. Responses get hashed and the first 4 KiB of the redacted response is written to `redacted_response_preview` for human triage.

### 3.5 `gateway.yaml` schema

```yaml
# gateway.yaml — opsbench-gateway configuration (loaded from $OPSBENCH_GATEWAY_CONFIG
# or ~/.config/opsbench/gateway.yaml). Reload via SIGHUP.

listen:
  address: "127.0.0.1:8765"          # default; bind to 0.0.0.0 only inside a network namespace
  tls:                                # optional; auto-cert via ACME later
    enabled: false

agent_class: "collector-security"     # the Cedar principal this gateway represents

policies:
  hand_written: "packages/team-incident-response/policies/cedar/tools.cedar"
  generated:    "packages/team-incident-response/policies/cedar/tools-generated.cedar"
  schema:       "packages/team-incident-response/policies/cedar/opsbench-schema.cedarschema.json"
  merge_order: ["hand_written", "generated"]   # hand_written wins

custody:
  path: "~/.local/share/opsbench/custody.log"
  rotate_at_bytes: 134217728          # 128 MiB
  keep_rotated: 8
  fsync_each_line: true               # default: durable; can be disabled for benchmark runs
  schema_version: "v1"

redaction_mode: "strict"
redaction_patterns:
  aws_access_key_id: '\bAKIA[0-9A-Z]{16}\b'
  bearer_token:      'Bearer\s+[A-Za-z0-9._\-]+'
  pem_block:         '-----BEGIN [A-Z ]+-----[\s\S]+?-----END [A-Z ]+-----'

rate_limit:
  per_second: 50
  burst: 100

upstreams:
  vault-mcp:
    transport: stdio
    command: ["vault-mcp-server", "--config", "/etc/vault-mcp/config.hcl"]
    env:
      VAULT_ADDR: "https://vault.internal:8200"
    upstream_timeout: 30s
    resource_shape:
      vault::kv::read:
        mount: "$.args.mount"
        path: "$.args.path"
    redaction:
      redact_keys: ["/args/data/password", "/args/data/token"]

  github-mcp:
    transport: streamable-http
    url: "http://localhost:9001/mcp"
    headers:
      Authorization: "Bearer ${GITHUB_MCP_TOKEN}"
    upstream_timeout: 30s

  grafana-mcp:
    transport: stdio
    command: ["mcp-grafana"]
    env:
      GRAFANA_URL: "https://grafana.internal"
      GRAFANA_API_KEY: "${GRAFANA_API_KEY}"
    upstream_timeout: 10s
```

The gateway templates `${VAR}` references at load time from the process environment; missing required vars fail-fast with a clear error.

### 3.6 Cedar evaluation contract

Inputs to every `Authorizer.IsAuthorized` call:

| field | source |
|-------|--------|
| `principal` | `User::"<gateway.yaml#agent_class>"` |
| `action` | `Action::"<upstream-id>::<tool-name>"` (e.g. `Action::"vault-mcp::vault::kv::read"`) |
| `resource` | `Resource::"<sha256-of-args-with-redaction>"`, attributes derived from `resource_shape` map |
| `context` | merged `{ human_approval, incident_round, incident_phase, verdict_status, tone_review_passed }` from request headers + `gateway.yaml#context_defaults` |

Decision = `allow` if any `permit` rule matches AND no `forbid` rule matches; `deny` otherwise. The evaluator records which rule matched (by Cedar `policy_id`) and that id flows into `custody.log#policy_files`. F1's hand-written `tools.cedar` is loaded first and its rules take precedence; ties go to `forbid`.

When a request's `action` is not declared in `opsbench-schema.cedarschema.json`, the evaluator returns `deny` with `deny_reason: "schema-unknown-action"`. This is the "schema closes the world" guarantee F1 establishes.

### 3.8 Pi-first integration

Per the F-series Pi-first mandate, every recipe leads with the Pi `AGENTS.md` block above. The gateway's URL form (`http://localhost:8765/mcp/<id>`) is identical across Pi and the other hosts; the only difference is config file location. The gateway emits a `pi-snippet` subcommand (`opsbench-gateway pi-snippet --upstream vault-mcp`) that prints the exact Pi AGENTS.md block for a configured upstream — recipe authors copy-paste the output into the recipe instead of hand-rolling it.

Pi has no native Cedar evaluator (Pi intentionally avoids built-in MCP — see F1 § 3.6 + parent roadmap § "Pi-first multi-host"). The gateway re-imposes the full Cedar policy server-side, so the Pi-side downshift in F1 (write-mode tools pushed to `human-operator`) survives: even when Pi grants a tool, the gateway can still `deny` based on context.

### 3.10 Vendoring vs sibling-repo decision

Open question for the reviewer (see § 7). Default in this spec: **vendor under `packages/opsbench-gateway/`** as a Go module rooted at that path. Rationale:

- Single repo / single PR pipeline keeps custody.log schema, Cedar policy, gateway code, and recipes in lockstep.
- Apache-2.0 → MIT compatibility is established (catalog § 1).
- Upstream-sync cadence is monthly; vendor copy is small (~30 kLOC Go).
- A sibling repo would require cross-repo PR coordination for every schema-version bump.

If the reviewer prefers sibling, the only code change is moving the Go module root; the design otherwise transfers without modification.

## 4. Cedar policy posture

F2 is the first phase that actually *evaluates* Cedar at runtime, so the policy posture here is concrete.

| Concern | F2 posture |
|---------|-----------|
| Default decision | **Deny.** No upstream is reachable until a matching `permit` rule resolves. |
| Policy source | Two files, merged: `tools.cedar` (hand, authoritative) + `tools-generated.cedar` (F1 generator). Both are validated against `opsbench-schema.cedarschema.json` at gateway startup; startup fails if either file is invalid. |
| Merge semantics | `tools.cedar` rules win on `(principal, action)` collisions. `forbid` rules always beat `permit` rules per Cedar semantics. |
| Schema closes the world | Any `action` not declared in the schema returns `deny` with `deny_reason: "schema-unknown-action"`. A new upstream that ships actions outside the schema fails fast. |
| Hot reload | `SIGHUP` re-loads policies + config. In-flight requests complete under the old policy; new requests use the new policy. A failed reload (validate error) keeps the previous policy active and logs `policy-reload-failed`. |
| Audit | Every decision (allow / deny / error) emits a custody.log line with `policy_files` listing matched rule IDs. |
| Privileged-tool gating | Cedar `permit` rules for `write`/`exec`/`mutate-with-undo` actions are guarded by `context.human_approval == true` (F1 contract). The gateway accepts the `X-Opsbench-Context-Human-Approval: true` header only when the request comes from `127.0.0.1` AND the in-flight TLS layer is bypassed for loopback OR the request carries a valid HMAC over the gateway's `~/.local/state/opsbench/human-approval.key` (a 32-byte secret materialised on first start, mode 0600). This stops a remote attacker from injecting the header even if the listen address is mistakenly bound to a routable interface. |
| Replay | A second identical request still requires a fresh evaluation (no in-memory result caching). Cedar evaluation is sub-millisecond per call; caching adds risk of stale-decision bugs without meaningful upside. |
| Policy provenance | At startup, gateway logs SHA-256 of each loaded policy file to custody.log as an `event_type: "gateway-start"` entry. Reviewers can confirm which policy version evaluated which decision by joining `gateway-start` to subsequent decisions on `gateway_id`. |
| F3 forward-compat | All decision fields (`decision`, `policy_files`, `args_sha256`, `response_sha256`) are the future inputs to F3's Ed25519 signature payload. F3 changes the *envelope* but not the inner fields. |

## 5. Acceptance criteria

1. `packages/opsbench-gateway/` exists, contains the vendored `stacklok/toolhive` source under `internal/upstream/toolhive/`, and ships a top-level `LICENSE` (Apache-2.0 from upstream) + `NOTICE` (toolhive copyright + opsbench changes) + `UPSTREAM` (pinned commit SHA + sync date).
2. `go build ./...` from `packages/opsbench-gateway/` produces a static binary on `linux/amd64`, `linux/arm64`, `darwin/arm64` (CI matrix).
3. `opsbench-gateway serve --config <fixture>` accepts the example `gateway.yaml`, fans out to three upstream MCPs simultaneously (fixture set: a fake stdio MCP + `containers/kubernetes-mcp-server` + `pab1it0/prometheus-mcp-server`), and round-trips a `tools/list` request from a test agent in <100ms p95.
4. Cedar evaluation:
   a. A `permit`-matched request returns the upstream response unmodified.
   b. A `forbid`-matched or default-deny request returns MCP error -32603 with `data.decision == "deny"`.
   c. An action not declared in the schema returns `deny` with `deny_reason == "schema-unknown-action"`.
   d. The integration test exercises all three paths.
5. Custody log:
   a. Every request emits exactly one JSON-Lines entry to `gateway.yaml#custody.path`.
   b. Entry validates against `schemas/custody-log.v1.json`.
   c. `args_sha256` and `response_sha256` are computed over canonicalised JSON (keys sorted, whitespace stripped).
   d. Redaction removes any key matching `redaction_patterns` or `redact_keys` before the entry is written to disk.
   e. Log rotation at `rotate_at_bytes` keeps `keep_rotated` files and rolls atomically (no torn entries).
6. CLI:
   a. `opsbench-gateway version` prints `<semver> (toolhive <upstream-sha>) cedar-go <version>`.
   b. `opsbench-gateway policy lint --config <gw.yaml>` validates both policy files against the schema and exits 0/2.
   c. `opsbench-gateway policy explain --action vault-mcp::vault::kv::write --principal collector-security` prints the matching `permit`/`forbid` rule(s) and the final decision.
   d. `opsbench-gateway custody verify-format <path>` validates an existing custody.log against the v1 schema, prints `OK` or a line-numbered failure.
   e. `opsbench-gateway pi-snippet --upstream vault-mcp` prints the Pi `AGENTS.md` config block for that upstream.
7. Distribution:
   a. `ghcr.io/shaiknoorullah/opsbench-gateway:<semver>` builds multi-arch on tag push, signed with cosign.
   b. GitHub Release attaches `opsbench-gateway-<os>-<arch>` binaries + SHA-256SUMS file.
   c. `docker run --rm ghcr.io/shaiknoorullah/opsbench-gateway:<semver> version` matches the binary's version output.
8. CI:
   a. `gateway-build.yml` runs on `pull_request` and `push` to main; matrix builds; uploads binary artifacts.
   b. `gateway-integration.yml` runs the three-upstream Compose test on every PR; fails if any acceptance criterion in (3)-(5) regresses.
   c. `gateway-build.yml` runs `go vet`, `staticcheck`, `golangci-lint`.
9. Recipe rewrites:
   a. Every F0 recipe gains a "Configuration — opsbench-gateway (recommended)" block as the first config block.
   b. The original direct-connection block is preserved under "Configuration — direct (advanced)" with the banner copy.
   c. The Pi `AGENTS.md` block leads each recipe's config section.
   d. `lint:md` continues to pass.
10. Documentation:
    a. `packages/opsbench-gateway/README.md` explains install (binary + Docker), config, the policy-merge story, and the custody-log shape.
    b. `packages/opsbench-gateway/UPSTREAM-SYNC.md` documents the monthly toolhive merge process.
    c. `docs/integrations.md` gains an opsbench-gateway entry under "Gateways" with a link to the README.
11. Single-host / single-tenant: gateway refuses to start if `listen.address` resolves to a non-loopback address unless `--allow-non-loopback` is passed (operator opts in). The startup log records the flag's presence.
12. Backwards compatibility: existing recipes that the user has *not* rewritten still function — agents continue to connect directly to upstreams. The gateway is an additive choke-point, never a replacement that breaks the direct path mid-incident.
13. PR shape: at most 5 PRs, each independently revertable:
    - PR 1: vendor toolhive + LICENSE/NOTICE/UPSTREAM + empty package shell.
    - PR 2: cedar evaluator + tests.
    - PR 3: custody log + redaction + tests.
    - PR 4: CLI surface + CI + Docker + Release pipeline.
    - PR 5: recipe rewrites (mechanical; lands last so a revert of PR 5 alone is enough to back the migration out).

## 6. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| `stacklok/toolhive` upstream pivots architecture (e.g. new policy hook surface) and our patches no longer apply cleanly | Medium | High | Pin a specific commit in `UPSTREAM`. Monthly merge with conflict-resolution doc. Patches live in `packages/opsbench-gateway/patches/` as quilt-style series; conflicts surface as failing CI on the sync PR rather than silent breakage. |
| `cedar-policy/cedar-go` has bugs that diverge from the Rust reference implementation used by F1's CLI | Medium | High | Add a differential test: for a fixture set of (policy, action, principal, context) tuples, run both `cedar` Rust CLI and `cedar-go` and assert identical decisions. Fixture file lives at `packages/opsbench-gateway/internal/cedar/testdata/diff-fixtures.json`. CI runs the diff weekly. |
| Custody-log corruption from concurrent writers | Low | High | Single writer goroutine; bounded channel from request handlers; `O_APPEND` open; per-line `fsync` (toggleable). Rotation acquires a file lock before swapping files. |
| Gateway bound to a routable interface leaks tool surface | Low | Critical | Default bind = `127.0.0.1`. Refusal to start on non-loopback without explicit `--allow-non-loopback`. The HMAC-signed `human_approval` header (§ 4) prevents privilege escalation even if the flag is set. |
| Upstream stdio MCPs hang forever | Medium | Medium | Per-upstream `upstream_timeout` (default 30s); on timeout, the gateway returns MCP error -32001 and writes `error_class: "upstream-timeout"` to custody. Hung child processes are killed via context cancellation. |
| Schema-version drift between F2's v1 and F3's v2 | Low | Medium | F3 schema bump is additive only (new fields, no rename / no semantic change). Gateway code in F2 already writes `schema_version: "v1"` literally so F3's reader can branch. |
| Redaction misses a secret pattern; secret hits custody.log | Medium | Critical | (a) Default `redaction_mode: strict` deny-list of common secret words. (b) Per-upstream `redact_keys`. (c) `opsbench-gateway custody verify-format --grep-secrets` subcommand scans an existing log for the redaction-pattern regexes and fails if matches are found; CI runs this against the integration-test custody.log. (d) Reviewer-facing docs spell out the policy-of-last-resort: "if your tool prints raw secrets in its `response`, set `upstream.redact_response_preview: false` so we hash but don't snippet". |
| Policy reload fails mid-incident and the gateway starts denying things | Low | High | Failed reload keeps previous policy; logs `policy-reload-failed`; exit code remains zero so process supervisors don't kill the gateway. `opsbench-gateway policy lint` runs as a `pre-reload` step in the supervisor unit. |
| Vendored toolhive's `LICENSE`/`NOTICE` falls out of sync with upstream | Low | Medium | Monthly sync PR updates both files; CI job `license-attribution-check.yml` diffs `LICENSE` against the pinned upstream commit. |
| Single-host bottleneck under high tool-call volume | Low (current usage) | Medium | F6 evaluates clustered or distributed deployment. F2 ships with the global `rate_limit` and the `/metrics` endpoint so operators can spot saturation early. |
| Pi `AGENTS.md` snippet format changes upstream | Medium | Low | `opsbench-gateway pi-snippet` is the single source of truth; recipe authors copy from CLI output. A Pi format change is one CLI patch + one regenerate-snippets pass. |
| Direct-connection block in recipes becomes the de facto path (users skip the gateway) | Medium | High | Each recipe's direct-connection block opens with the banner ("Bypasses opsbench Cedar evaluation + custody.log; F3 receipts will not cover direct calls"). `scripts/doctor.sh` checks that the user's recipe configs reference `localhost:8765` and warns when they don't. |

## 7. Open questions for reviewer

1. **Vendor vs sibling repo.** Default in this spec is vendoring under `packages/opsbench-gateway/` (rationale in § 3.10). The reviewer may prefer a sibling repo for clearer Apache-2.0 vs MIT boundaries. If sibling, the only knock-on change is moving the Go module root and adding a submodule pin under `packages/opsbench-gateway/upstream/`.
2. **`cedar-go` vs invoking the Rust CLI via subprocess.** `cedar-go` (in-process) is faster (~50µs/eval) and simpler; the Rust CLI is the reference. Default: `cedar-go` with the differential test (§ 6). Alternative: in-process via `cgo` against `cedar-policy-cli`. The reviewer's call decides whether we hard-pin the Rust ABI as a build-time dep.
3. **`policy reload` on file change vs SIGHUP only.** SIGHUP is operator-explicit and safe but requires a process-supervisor recipe per host. File-watcher reload (fsnotify) is more ergonomic but risks reloading mid-edit. Default: SIGHUP only. Reviewer may want file-watcher for development convenience.
