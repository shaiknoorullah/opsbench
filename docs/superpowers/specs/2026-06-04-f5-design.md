## 1. Purpose

F5 is the headline distribution phase. Today opsbench's first-class host is Claude Code, with Codex CLI handled by a best-effort adapter that leaves `TODO_AGENT_CALL`, `TODO_TASK_CREATE`, and `TODO_SKILL_INVOKE` placeholders in the output and no marketplace presence anywhere. The strategic shift in F5 is to make **Pi (pi.dev)** the primary host opsbench targets, with Claude Code, Codex CLI, GitHub Copilot, Cursor, Gemini, and OpenCode brought to parity through generated adapters. Pi is intentionally a minimal terminal coding harness without built-in MCP, which means opsbench must integrate via Pi's actual surfaces: CLI tools the user installs and Pi shells out to via `Bash`, Pi extensions installed via `pi install npm:@scope/pkg` or `pi install git:github.com/owner/repo`, project-level `AGENTS.md` files, per-project `SYSTEM.md` customization, and `models.json` provider/model registration. Authoring the canonical Pi distribution of the 11 skills + 33 agents already in `team-incident-response` is the substantive engineering work; the multi-host adapters and the installer matrix (Homebrew tap, AUR PKGBUILD, `flake.nix`) ride on top of that foundation. The strategic positioning win is that opsbench becomes the de-facto ops/SRE toolkit for Pi users, which is a much narrower and less-contested market segment than "Claude Code ops skills" — and the installer matrix turns a one-host project into a cross-distro, cross-host one without forking opsbench's source layout.

## 2. Scope

### In-scope — Pi-first work (the headline)

- **New top-level adapter directory.** `tools/pi-compat-layer/` mirroring the existing `tools/codex-compat-layer/`. Owns the Pi-side translation layer.
- **Generator script.** `tools/pi-compat-layer/adapt.sh` reads every opsbench `SKILL.md` and the per-skill `AGENT.md` files, and emits a Pi-native skill manifest tree under `${PI_PREFIX:-${HOME}/.pi}/skills/<team>/<skill>/`. Each output tree contains: `manifest.json` (Pi skill manifest), `SKILL.md` (Pi-flavored markdown — Skill/Agent/TaskCreate replaced with Pi-native equivalents, not TODO placeholders), `AGENTS.md` fragment (appended to the project's root `AGENTS.md` during install), and optional `hooks/` (see safety hooks below).
- **Canonical Pi distribution of all 11 skills + 33 agents.** Authored once, generated for every other host. The skills are: `evidence-analyze`, `evidence-cataloger`, `evidence-collection-orchestrator`, `evidence-request`, `evidence-source-discovery`, `forensic-synthesis`, `incident-quarantine`, `incident-timeline`, `parallel-hypothesis-debug`, `post-incident-artifact-generator`, `storage-incident-response`. The agents live under `packages/team-incident-response/agents/team-{1..8}-*/` and total 33 files.
- **AGENTS.md generation.** Pi reads project-level `AGENTS.md` as a system prompt. The adapter generates `${PI_PREFIX}/agent/AGENTS.md` from each opsbench skill's frontmatter `description` field plus a short "when to use" block. A per-project `AGENTS.md` is generated under `packages/team-incident-response/AGENTS.md` so anyone running Pi inside the opsbench checkout gets the right system prompt automatically.
- **models.json provisioning.** `tools/pi-compat-layer/models.json.template` ships a default `models.json` registering the Anthropic Claude provider (with `claude-opus-4-7` as default), the Codex provider (so `pi --provider codex` works), and an OpenAI provider stub. `tools/pi-compat-layer/install-models.sh` merges this template into the user's existing `~/.pi/models.json` non-destructively (preserving any custom providers).
- **Pi-specific safety hooks.** `tools/pi-compat-layer/hooks/` ships shell scripts adapting opsbench's `PreToolUse` / `PostToolUse` / `SubagentStop` conventions to Pi's hook conventions. Mapping is documented in section 5 below. Pi's hook execution model is shell-based (similar to Claude Code) so the adaptation is mostly a rename + path rewrite rather than a rewrite.
- **Pi marketplace registration.** Submit opsbench as a Pi plugin to the Pi plugin marketplace. Submission artifact lives at `tools/pi-compat-layer/marketplace/manifest.yaml`. Approval is async; the PR ships the submission, and the README documents the install path while approval is pending.
- **`pi-validate` CI job.** `.github/workflows/pi-validate.yml` runs the adapter on every PR that touches `packages/team-incident-response/**`, `tools/pi-compat-layer/**`, or the F5 specs, and validates the generated manifests against `tools/pi-compat-layer/schemas/manifest.schema.json` plus a smoke-test that runs `pi --dry-run skill list` inside the generated layout.

### In-scope — Other-host parity (after Pi lands)

- **Codex adapter hardening.** `tools/codex-compat-layer/adapt.sh` today leaves `TODO_AGENT_CALL`, `TODO_TASK_CREATE`, and `TODO_SKILL_INVOKE` strings in the output. F5 replaces those with real Codex equivalents: `TODO_AGENT_CALL` → an inline workflow that spawns a second `codex` subprocess with a constrained prompt; `TODO_TASK_CREATE` → a `TODO.md` planning-file pattern with a follow-up reminder injected into the Codex session; `TODO_SKILL_INVOKE` → inlined skill body with a header reminding Codex this was originally a slash-invocation.
- **Auto-emit variants for every host.** New per-host adapter directories: `tools/copilot-compat-layer/`, `tools/cursor-compat-layer/`, `tools/gemini-compat-layer/`, `tools/opencode-compat-layer/`. Each contains a `README.md` documenting the install flow and an `adapt.sh` modeled on the codex one. The Pi adapter is the source of truth; the other adapters are projections.
- **Per-host CI validation jobs.** `.github/workflows/codex-validate.yml`, `copilot-validate.yml`, `cursor-validate.yml`, `gemini-validate.yml`, `opencode-validate.yml`. Each runs its host's `adapt.sh --all` and verifies the output tree is non-empty, the manifests parse, and no `TODO_*` placeholder strings remain in any generated file.
- **Per-host README.** `tools/<host>-compat-layer/README.md` documents the install command (`bash scripts/install.sh --host <host>`), the output location (`${HOST_PREFIX}/skills/`), and the known caveats.

### In-scope — Installer matrix (last in F5)

- **Homebrew formula + tap.** New tap `shaiknoorullah/opsbench` at `github.com/shaiknoorullah/homebrew-opsbench`. Formula `Formula/opsbench.rb` installs the opsbench tree under `${HOMEBREW_PREFIX}/share/opsbench/` and symlinks the host-specific outputs (`~/.pi/skills/...`, `~/.claude/skills/...`, `~/.codex/skills/...`) by default to Pi-first layout. CLI flag override: `brew install opsbench --with-host=claude-code` switches the primary symlink target.
- **AUR PKGBUILD.** `tools/installer/aur/opsbench/PKGBUILD` packages opsbench for Arch Linux's AUR. Default `install` script honors the same Pi-first layout. Package name `opsbench`; meta-package `opsbench-all` pulls every host adapter. Trusted-User submission lives at `tools/installer/aur/SUBMISSION.md`.
- **Nix flake.** `flake.nix` at repo root exposes `packages.default = opsbench` (Pi-first), `packages.opsbench-claude` (Claude-Code-first), `packages.opsbench-codex` (Codex-first), and `apps.default` so `nix run github:shaiknoorullah/opsbench` works without a local clone.
- **CI matrix on tag push.** `.github/workflows/release-installers.yml` on `v*` tags runs three parallel jobs: `brew-bottle` (builds a Homebrew bottle for darwin-arm64 + linux-x86_64), `aur-publish` (pushes the new PKGBUILD to the AUR via SSH), `nix-build` (builds the flake on `nix-build` to verify reproducibility). Failures gate the GitHub Release publish step.

### Out-of-scope (preserve foundation-first sequencing)

- **Cedar policy regeneration.** F1 owns the Cedar generator. F5 only re-projects `tools-generated.cedar` into Pi's allowlist shape. If F1 changes the schema, F5 picks it up.
- **Gateway behavior changes.** F2 owns `opsbench-gateway` semantics. F5 ships gateway URL examples in Pi config docs but does not modify gateway code.
- **Receipt schema or signing.** F3 owns the receipt schema and key bootstrap. F5 ships the verifier in the installer payload; it does not change the schema.
- **New team packages.** F4 is in flight in parallel. F5 only operates on `team-incident-response` because that is the only team with skills + agents at the time F5 begins. Once an F4 team merges, a follow-up "retrofit" PR re-runs the Pi adapter against it.
- **Windows packaging.** No `winget`, `scoop`, or `choco` package in F5. The Homebrew + AUR + nix triple covers the realistic ops/SRE distro footprint. Windows users install via the manual `scripts/install.sh --host <host>` path.
- **Pi runtime changes.** opsbench does not patch or vendor Pi. If Pi's manifest schema changes, the adapter changes; Pi itself is unmodified.
- **Multi-user marketplace presence beyond Pi.** Cursor, Gemini, OpenCode do not yet have stable plugin marketplaces matching opsbench's shape. F5 ships per-host adapters, not per-host marketplace submissions, for any host other than Pi.

## 3. Architecture & data flow

### 3.1 Directory layout under `tools/pi-compat-layer/`

```text
tools/pi-compat-layer/
├── README.md                          # install flow + caveats (mirrors codex README)
├── adapt.sh                           # main generator: SKILL.md → Pi manifest tree
├── install-models.sh                  # non-destructive merge into ~/.pi/models.json
├── models.json.template               # default provider registrations
├── marketplace/
│   └── manifest.yaml                  # Pi plugin marketplace submission artifact
├── schemas/
│   └── manifest.schema.json           # JSONSchema for the Pi manifest output
├── hooks/
│   ├── pre-tool-use.sh                # adapted from opsbench's PreToolUse
│   ├── post-tool-use.sh               # adapted; emits custody.log + verifies receipts
│   └── subagent-stop.sh               # adapted from SubagentStop
└── templates/
    ├── AGENTS.md.tmpl                 # per-project AGENTS.md skeleton
    ├── SYSTEM.md.tmpl                 # per-project SYSTEM.md skeleton
    └── skill-manifest.json.tmpl       # Pi skill manifest skeleton
```

### 3.2 Adapter pipeline (SKILL.md → Pi manifest)

```text
packages/team-incident-response/skills/<skill>/SKILL.md
    │
    │  (parse frontmatter via yq; extract description, tools, allowed_tools, hooks)
    ▼
tools/pi-compat-layer/adapt.sh
    │
    │  (template render: skill-manifest.json.tmpl + body rewrites)
    │  - replace `Skill(...)` invocations with `pi skill run <name>`
    │  - replace `Agent(...)` calls with `pi spawn --skill <subagent>`
    │  - replace `TaskCreate(...)` with `pi task add` CLI calls
    │  - leave `Bash`, `Read`, `Edit`, `Write`, `Grep`, `Glob` unchanged
    │  - rewrite `mcp__*` references to point at $OPSBENCH_GATEWAY_URL
    ▼
${PI_PREFIX:-${HOME}/.pi}/skills/<team>/<skill>/
    ├── manifest.json                   # validated against schemas/manifest.schema.json
    ├── SKILL.md                        # Pi-flavored body
    ├── AGENTS.md                       # appended to ~/.pi/agent/AGENTS.md on install
    └── hooks/                          # copied from tools/pi-compat-layer/hooks/ if skill declares any
```

### 3.3 AGENTS.md generation

Pi reads `~/.pi/agent/AGENTS.md` as the global system prompt and per-project `AGENTS.md` files as scoped overrides. The adapter:

1. Reads every skill's frontmatter `description` field.
2. Concatenates `<!-- begin opsbench/<skill> -->` ... `<!-- end opsbench/<skill> -->` blocks containing a 2–4 line "when to use" excerpt plus the skill's invocation command.
3. Writes the result to `${PI_PREFIX}/agent/AGENTS.md` and to `packages/team-incident-response/AGENTS.md` (project-scoped).
4. Existing user content between `<!-- begin opsbench:user -->` and `<!-- end opsbench:user -->` markers is preserved across regeneration.

### 3.4 models.json provisioning

`tools/pi-compat-layer/models.json.template` declares:

```json
{
  "providers": {
    "anthropic": {
      "endpoint": "https://api.anthropic.com",
      "default_model": "claude-opus-4-7",
      "auth_env": "ANTHROPIC_API_KEY"
    },
    "codex": {
      "command": "codex",
      "default_model": "gpt-5-codex"
    },
    "openai": {
      "endpoint": "https://api.openai.com/v1",
      "default_model": "gpt-5",
      "auth_env": "OPENAI_API_KEY"
    }
  }
}
```

`install-models.sh` merges this template into `~/.pi/models.json` with `jq`, preserving any user-added providers. If the user has already declared a conflicting key, the script prints a diff and exits non-zero unless `--force` is passed.

### 3.5 Hook adaptation table

opsbench skills declare hooks via frontmatter. The Pi adapter maps each opsbench hook to a Pi hook with identical semantics. The mapping is intentionally symmetric so custody-log payloads are byte-identical across hosts and the F3 verifier can validate them without host-aware parsing.

| opsbench hook       | Pi equivalent                            | Trigger point                                  | Notes |
| ------------------- | ---------------------------------------- | ---------------------------------------------- | ----- |
| `PreToolUse`        | Pi `pre-exec` hook (registered in `~/.pi/hooks/pre-exec`) | Before any `Bash`, `Read`, `Edit`, `Write`, or `mcp__*` call | Evaluates Cedar policy; aborts the call on `Deny`. Receives JSON on stdin with `tool`, `args_sha256`, `agent_class`. |
| `PostToolUse`       | Pi `post-exec` hook (`~/.pi/hooks/post-exec`)            | After every tool call returns                  | Appends a custody-log entry (`schemas/custody-log.v1.json` from F2). On Linux, verifies the F3 Ed25519 receipt if present. |
| `SubagentStop`      | Pi `task-complete` hook (`~/.pi/hooks/task-complete`)    | When a Pi `spawn` finishes (success or failure) | Captures subagent outcome + final state into custody.log under `event: subagent_stop`. |
| `SessionStart`      | Pi `session-start` hook                                  | New Pi session begins                          | Sources `~/.config/opsbench/env` to ensure `$OPSBENCH_GATEWAY_URL` is exported. |
| `Stop` (Claude Code only) | not mapped (no Pi equivalent)                      | n/a                                            | Custody emission for session-end is handled by `SubagentStop` on the outermost agent. |

## 4. Cedar policy posture

F5 adds no new Cedar rules. It re-projects the F1 output into Pi-readable form so Pi can honor the same allowlist decisions at the host level when running offline (without `opsbench-gateway` in the path).

- The Pi adapter reads `packages/team-incident-response/policies/cedar/tools-generated.cedar` (from F1) plus the authoritative `tools.cedar` override.
- It uses `tools/pi-compat-layer/cedar-emit.sh` (introduced in F1 §2 in-scope list, populated in F5) to convert the Cedar AST into Pi's allowlist manifest format. The conversion is lossy for `when` clauses that reference Cedar context attributes Pi cannot evaluate; those rules degrade to `deny-by-default` and the adapter prints a warning per dropped rule.
- The generated Pi allowlist lands at `${PI_PREFIX}/allowlists/opsbench.json` and is referenced from the Pi `pre-exec` hook so every tool call is evaluated against the same allowlist that `opsbench-gateway` enforces.
- When the gateway is in the path (F2 default), Cedar evaluation happens at the gateway and the Pi-side allowlist is treated as a belt-and-suspenders second check, not the primary control.
- The F3 receipt verifier (`scripts/verify-receipts.sh`) is shipped unchanged in every installer payload. It is a no-op on hosts where the gateway is not configured.

## 5. Acceptance criteria

1. `tools/pi-compat-layer/adapt.sh` exists, is executable, and accepts `--all`, `--team <slug>`, and `<path-to-skill.md>` modes mirroring `tools/codex-compat-layer/adapt.sh`.
2. Running `bash tools/pi-compat-layer/adapt.sh --all` against the current repo emits a manifest tree for all 11 skills + 33 agents under `${PI_PREFIX:-${HOME}/.pi}/skills/team-incident-response/`.
3. `.github/workflows/pi-validate.yml` exists and passes: validates 11 skill manifests + 33 agent manifests against `tools/pi-compat-layer/schemas/manifest.schema.json` and confirms zero `TODO_*` placeholder strings remain.
4. `tools/pi-compat-layer/marketplace/manifest.yaml` exists and is submitted to the Pi marketplace; the README documents the install path while approval is pending.
5. `tools/codex-compat-layer/adapt.sh` no longer emits `TODO_AGENT_CALL`, `TODO_TASK_CREATE`, or `TODO_SKILL_INVOKE`; `codex-validate` CI job passes on the entire `packages/team-incident-response/` tree.
6. Adapter directories `tools/copilot-compat-layer/`, `tools/cursor-compat-layer/`, `tools/gemini-compat-layer/`, and `tools/opencode-compat-layer/` exist with `adapt.sh` + `README.md`, and their respective `*-validate` CI jobs pass.
7. `tools/installer/homebrew/opsbench.rb` exists, and `brew install shaiknoorullah/opsbench/opsbench` installs successfully on macOS (darwin-arm64) and Linux (linux-x86_64) with Pi-first layout as the default symlink target.
8. `tools/installer/aur/opsbench/PKGBUILD` exists, and `yay -S opsbench` (or `paru -S opsbench`) installs successfully on Arch Linux.
9. `flake.nix` at repo root exists, and `nix run github:shaiknoorullah/opsbench` runs the default Pi-first installer end-to-end with no manual intervention.
10. Every team package's `README.md` (today: `packages/team-incident-response/README.md`; later: F4 team READMEs as they ship) lists supported hosts in the priority order from §3.6.
11. `tools/pi-compat-layer/install-models.sh` merges its template into `~/.pi/models.json` without overwriting any user-added provider entry, and exits non-zero with a diff on conflict unless `--force` is passed.
12. The Pi pre-exec hook (`tools/pi-compat-layer/hooks/pre-tool-use.sh`) reads `${PI_PREFIX}/allowlists/opsbench.json` and denies every tool call not present in the allowlist; an integration test in `pi-validate` confirms a synthetic denied call exits with the expected error.
13. `.github/workflows/release-installers.yml` runs `brew-bottle`, `aur-publish`, and `nix-build` jobs in parallel on every `v*` tag, and a release cannot publish if any of the three fail.
14. `scripts/install.sh --host pi` runs end-to-end on a fresh Linux container in under 90 seconds and leaves `${PI_PREFIX}` populated with skills, agents, allowlist, hooks, and `models.json`.
15. The F3 receipt verifier `scripts/verify-receipts.sh` is included in every installer payload (Homebrew, AUR, Nix) and validates a sample `custody.log` against a fixture public key as part of the installer smoke test.

## 6. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
| ---- | ---------- | ------ | ---------- |
| Pi manifest schema changes between adapter authoring and Pi marketplace approval | medium | high (every adapted skill becomes invalid) | Pin to the Pi schema version recorded in `tools/pi-compat-layer/schemas/manifest.schema.json`; monitor Pi's CHANGELOG via `watch` skill; add an explicit `pi-schema-version` field in `manifest.json` so the validator fails loud on mismatch. |
| Pi marketplace approval lag (weeks → months) | high | medium | Ship the install-from-source path (`pi install git:github.com/shaiknoorullah/opsbench`) in F5; do not gate F5 release on marketplace approval. Document the marketplace status in the README. |
| Codex placeholder semantics still unclear after F5 (e.g., `Agent` has no real Codex equivalent) | medium | medium | Treat the Codex replacement as best-effort: spawn-subprocess-per-agent works for stateless agents; for stateful ones (`parallel-hypothesis-debug`), the adapter emits a documented manual fallback at the top of the file and `codex-validate` whitelists those specific skills from the no-`TODO_*` check. |
| Homebrew tap maintenance burden (every tag needs a tap PR) | medium | low | Automate the tap push from `.github/workflows/release-installers.yml`; the tap repo is generated, not hand-authored. Tap README is regenerated from the source tap-README.md on every release. |
| AUR review delays + TU contact friction | medium | low | Submit to AUR under the maintainer's own account first (AUR allows unprivileged submissions of new packages); pursue Trusted-User adoption only if the package becomes popular enough to warrant it. |
| Nix flake reproducibility breaks across nixpkgs revisions | medium | medium | Pin `nixpkgs` revision in `flake.lock`; run `nix flake check` in CI on every PR touching `flake.nix`; document the supported `nixpkgs` revision range in `tools/installer/nix/README.md`. |
| Per-host adapter explosion (5 adapters × every skill change × every team package = O(N×M) work) | high | medium | Pi adapter is the source of truth; the other 5 are projections generated by their `adapt.sh` from the Pi output. A skill author edits Claude-Code-native files; CI ensures all 6 host outputs stay in sync. Adapter scripts share a common library at `tools/_lib/adapt-common.sh`. |
| Hook adaptation breaks custody.log byte-for-byte parity (F3 receipt verification fails on Pi-emitted logs) | low | high | Hook scripts share the same payload-construction code at `tools/_lib/custody-payload.sh`; the only host-specific code is the trigger registration. `pi-validate` includes a parity test that runs a known-input scenario on both Claude Code and Pi and asserts identical custody.log lines (modulo timestamp). |

## 7. Open questions for reviewer

1. **Marketplace gating.** Should F5 release be allowed to merge before the Pi marketplace approval lands? Default in this spec: yes — install-from-source covers the gap. Reviewer may prefer to hold the release tag until marketplace approval lands so the README does not advertise an unapproved listing.
2. **Codex hard-case carve-out.** `parallel-hypothesis-debug` and `forensic-synthesis` fundamentally rely on Claude Code's `Agent` primitive. Should `codex-validate` whitelist those two skills from the no-`TODO_*` check (this spec's default), or should F5 ship a real workaround (e.g., `tmux` session multiplexing in the adapter)?
3. **Installer-matrix scope.** Should F5 ship Windows packaging (winget) as a stretch goal, or hold it for a future F-phase? Default in this spec: hold. Reviewer may want to expand if Windows ops teams are an explicit target.
