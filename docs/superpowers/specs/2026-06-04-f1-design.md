- F0 design (in flight at the time of this doc): the bulk-ship of ~30 MCP recipes whose `tools:` frontmatter blocks F1 consumes.

## 1. Purpose

Today, every per-agent Cedar allowlist in `packages/team-incident-response/policies/cedar/tools.cedar` is hand-written. That works for the 33 baseline subagents but does not scale to the ~30 new MCP recipes shipping in F0, the five new team packages shipping in F4, and the Pi-first multi-host parity work in F5. F1 adopts `cedar-policy/cedar-for-agents` as the conversion layer that reads each recipe's `tools:` frontmatter block, derives a least-privilege Cedar allowlist per agent class, and writes the merged result to a single generated policy file. The hand-written file stays as the authoritative override (hand rules beat generated rules at evaluation time), so reviewers retain veto power on every decision the generator makes. This phase produces no runtime gateway code — it is a build-time + CI-time tooling phase that prepares the ground for F2's gateway and F3's signed receipts.

## 2. Scope

### In-scope

- A new generator script: `scripts/generate-cedar-policy.sh`. Bash wrapper around the `cedar-for-agents` CLI; reads recipe frontmatter, emits one consolidated Cedar file.
- A new output artifact: `packages/team-incident-response/policies/cedar/tools-generated.cedar`. Gitignored; regenerated on `pnpm install` and in CI.
- A new structured frontmatter contract: every recipe MUST have a `tools:` block declaring (a) the list of MCP tool names the recipe surfaces, (b) the read/write classification per tool, (c) the suggested agent classes that should be granted access, (d) the resource shape constraints. Recipes without the block are skipped (no error, only a warning) so backfill can land incrementally.
- Backfill: add the `tools:` block to the 16 pre-existing recipes under `packages/team-incident-response/mcp-recipes/`. Eight of these are inherited from old P1 work (`thehive-mcp.md`, `opencti-mcp.md`, `azure-skills-mcp.md`, `k8sgpt-mcp.md`, etc., per the F-series roadmap), and the remainder are the older catalog set (`aws-mcp.md`, `clickhouse-mcp.md`, `ebpf-observability-mcp.md`, `k8s-mcp.md`, `linear-mcp.md`, `opentelemetry-mcp.md`, `pagerduty-mcp.md`, `postgres-mcp.md`, `slack-mcp.md`, `velociraptor-mcp.md`, `azure-mcp.md`, `github-mcp.md`, `grafana-mcp.md`, plus three `CUSTOM-*` recipes).
- A new CI job: `validate-cedar`. Runs the generator in a clean checkout, diffs the output against a snapshot, and validates both files with `cedar validate` from the official Cedar CLI.
- A new `prepare` script entry in the root `package.json` so the generator runs on local `pnpm install` (idempotent; no-op if the recipes haven't changed since the last run).
- A small adapter, `tools/pi-compat-layer/cedar-emit.sh`, that converts the generated Cedar file into Pi's expected manifest shape (per F5's Pi-first mandate). This stays in F1 to avoid Pi having a half-supported policy story when F2 ships.

### Out-of-scope

- Runtime evaluation. The F2 `opsbench-gateway` consumes both Cedar files at request time; nothing in F1 stands up the gateway.
- Ed25519-signed receipts. F3 owns that schema and the key bootstrap.
- New recipes beyond what F0 ships. If F0 trims the recipe count from ~38 down to ~30, F1 simply iterates fewer files.
- Generating per-host (Claude Code / Codex / Copilot / Cursor / Gemini / OpenCode) variants of the Cedar policy. F5 owns multi-host emission. F1 produces the canonical Cedar file plus the Pi adapter only.
- Rewriting the existing 617-line `tools.cedar`. It stays untouched in F1 (it is the override). A follow-up cleanup in F4 will identify which hand-rules became redundant once `tools-generated.cedar` lands and prune them.
- Policy-decision telemetry. The `custody.log` work in F2 covers per-decision logging; F1 only verifies syntactic validity.

### 3.1 Recipe `tools:` frontmatter contract

Each recipe gains a YAML block at the top of the markdown file (delimited by `---`). Example for the F0 `vault-mcp.md` recipe:

```yaml
---
recipe_id: vault-mcp
upstream: hashicorp/vault-mcp-server
license: MPL-2.0
status: stable
last_verified: 2026-06-04
tools:
  - name: vault::kv::read
    mode: read
    resource_shape:
      mount: string
      path: string
    suggested_agent_classes:
      - collector-security
      - evidence-source-discoverer
    notes: KV v1 and v2 supported; path templates allowed.
  - name: vault::kv::write
    mode: write
    resource_shape:
      mount: string
      path: string
    suggested_agent_classes:
      - human-operator
    requires_human_approval: true
  - name: vault::pki::issue
    mode: write
    resource_shape:
      mount: string
      common_name: string
    suggested_agent_classes:
      - human-operator
    requires_human_approval: true
---
```

Rules:

1. `name` MUST be a Cedar-action-safe identifier (alphanumeric + `::` + `_`). The generator validates this.
2. `mode` MUST be one of `read`, `write`, `exec`, `mutate-with-undo`. `read` produces `permit` rules; `write`, `exec`, and `mutate-with-undo` produce `permit` rules guarded by `context.human_approval == true` unless the recipe is explicitly marked `safe_writes: true`.
3. `suggested_agent_classes` MUST reference existing agent classes from `packages/team-incident-response/agents/*.md` (the generator cross-checks against the agent directory and warns on unknown classes). The list `["human-operator"]` is always permitted.
4. `requires_human_approval: true` is a redundant explicit marker for write/exec modes; it always generates the `human_approval == true` guard. The redundancy is intentional — it makes recipes self-documenting without forcing reviewers to know the mode defaults.
5. Tool entries can omit `resource_shape` for tools that have no resource (e.g., `version::get`). The generator emits an unconstrained `resource` clause in that case.

A sibling file `tools.yaml` is also acceptable for recipes whose markdown content is too long for editors that fold the frontmatter. The generator prefers the frontmatter when both are present.

### 3.2 Generator script

`scripts/generate-cedar-policy.sh`:

```bash
#!/usr/bin/env bash
# F1 — Cedar policy generator. Reads recipe frontmatter, emits tools-generated.cedar.
set -euo pipefail

OPSBENCH_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
RECIPES_DIR="${OPSBENCH_ROOT}/packages/team-incident-response/mcp-recipes"
POLICIES_DIR="${OPSBENCH_ROOT}/packages/team-incident-response/policies/cedar"
SCHEMA_PATH="${POLICIES_DIR}/opsbench-schema.cedarschema.json"
TEMPLATE_PATH="${POLICIES_DIR}/templates/agent-class.cedar.tmpl"
OUTPUT_PATH="${POLICIES_DIR}/tools-generated.cedar"
WORK_DIR="$(mktemp -d)"
trap 'rm -rf "$WORK_DIR"' EXIT

# Tooling:
#   - yq (mikefarah/yq v4) for frontmatter parsing
#   - cedar-for-agents CLI: cargo-installed binary `cedar-for-agents`
#   - cedar CLI for validation
command -v yq >/dev/null || { echo "ERR: yq not installed (brew install yq)"; exit 2; }
command -v cedar-for-agents >/dev/null || { echo "ERR: cedar-for-agents missing; run scripts/install-cedar-tools.sh"; exit 2; }
command -v cedar >/dev/null || { echo "ERR: cedar missing; run scripts/install-cedar-tools.sh"; exit 2; }

# 1. Walk every recipe; extract the `tools:` block.
TOOLS_JSON="${WORK_DIR}/tools.json"
echo "[]" > "$TOOLS_JSON"
for recipe in "$RECIPES_DIR"/*.md; do
  [[ "$(basename "$recipe")" == README* ]] && continue
  frontmatter="$(awk '/^---$/{c++; next} c==1{print}' "$recipe")"
  [[ -z "$frontmatter" ]] && { echo "skip (no frontmatter): $recipe" >&2; continue; }
  block="$(printf '%s\n' "$frontmatter" | yq -o=json '. | {recipe_id, tools}')"
  jq -e '.tools' >/dev/null 2>&1 <<<"$block" || { echo "skip (no tools): $recipe" >&2; continue; }
  TOOLS_JSON_TMP="${WORK_DIR}/_tmp.json"
  jq --argjson r "$block" '. + [$r]' "$TOOLS_JSON" > "$TOOLS_JSON_TMP"
  mv "$TOOLS_JSON_TMP" "$TOOLS_JSON"
done

# 2. Cross-check agent classes exist.
AGENT_LIST="${WORK_DIR}/agents.txt"
ls "${OPSBENCH_ROOT}/packages/team-incident-response/agents"/*.md 2>/dev/null \
  | xargs -n1 basename \
  | sed 's/\.md$//' \
  > "$AGENT_LIST"
UNKNOWN="$(jq -r '[.[].tools[].suggested_agent_classes[]] | unique | .[]' "$TOOLS_JSON" \
  | grep -vxFf "$AGENT_LIST" \
  | grep -vx "human-operator" || true)"
if [[ -n "$UNKNOWN" ]]; then
  echo "WARN: unknown agent classes referenced in recipes:" >&2
  echo "$UNKNOWN" >&2
fi

# 3. Invoke cedar-for-agents.
cedar-for-agents convert \
  --input "$TOOLS_JSON" \
  --schema "$SCHEMA_PATH" \
  --template "$TEMPLATE_PATH" \
  --emit-cedar \
  > "$OUTPUT_PATH.new"

# 4. Validate.
cedar validate \
  --schema "$SCHEMA_PATH" \
  --policies "$OUTPUT_PATH.new"

# 5. Atomic install.
mv "$OUTPUT_PATH.new" "$OUTPUT_PATH"
echo "[ok] wrote $OUTPUT_PATH (recipes: $(jq 'length' "$TOOLS_JSON"))"
```

Notes:

- The script is dependency-light: `yq`, `jq`, `cedar`, `cedar-for-agents`. A companion `scripts/install-cedar-tools.sh` shells out to `cargo install cedar-for-agents` (pinned) and `cargo install cedar-policy-cli` (pinned). The installer is idempotent and skips already-installed binaries.
- The `--template` file at `policies/cedar/templates/agent-class.cedar.tmpl` is a hand-edited Cedar fragment (Handlebars-style placeholders) that lets us shape what generated rules look like — keeps the output stylistically consistent with `tools.cedar` so diffs read cleanly.
- The script is intentionally non-parallel. ~38 recipes is too small to parallelize, and serial execution makes the warnings ordering deterministic for snapshot tests.

### 3.3 Output shape (`tools-generated.cedar`)

Sample generated rule (from the `vault-mcp.md` example above):

```cedar
// Generated by scripts/generate-cedar-policy.sh from packages/team-incident-response/mcp-recipes/vault-mcp.md.
// DO NOT EDIT BY HAND. Override in packages/team-incident-response/policies/cedar/tools.cedar instead.

permit (
    principal in [
        User::"collector-security",
        User::"evidence-source-discoverer"
    ],
    action == Action::"vault::kv::read",
    resource
);

permit (
    principal == User::"human-operator",
    action == Action::"vault::kv::write",
    resource
) when {
    context.human_approval == true
};

permit (
    principal == User::"human-operator",
    action == Action::"vault::pki::issue",
    resource
) when {
    context.human_approval == true
};
```

The generated file is appended to `.gitignore`. Consumers (the F2 gateway, CI, the doctor script) regenerate it on demand.

### 3.4 Schema file (`opsbench-schema.cedarschema.json`)

The schema enumerates every action namespace the generator is allowed to emit. New recipe authors who need a new namespace must add it here in the same PR as their recipe — this is the gatekeeping mechanism that prevents arbitrary action proliferation. Sketch:

```jsonc
{
  "Opsbench": {
    "entityTypes": {
      "User": { "shape": { "type": "Record", "attributes": {} } },
      "Resource": {
        "shape": {
          "type": "Record",
          "attributes": {
            "path": { "type": "String", "required": false },
            "namespace": { "type": "String", "required": false },
            "node_role": { "type": "String", "required": false },
            "mount": { "type": "String", "required": false },
            "read_only": { "type": "Boolean", "required": false }
          }
        }
      }
    },
    "actions": {
      // K8s family
      "k8s::get": { "appliesTo": { "principalTypes": ["User"], "resourceTypes": ["Resource"] } },
      "k8s::list": { "appliesTo": { "principalTypes": ["User"], "resourceTypes": ["Resource"] } },
      "k8s::scale": { "appliesTo": { "principalTypes": ["User"], "resourceTypes": ["Resource"] } },
      // ... (one entry per namespaced action the existing tools.cedar already references)
      // F0/F1 additions:
      "vault::kv::read":  { "appliesTo": { "principalTypes": ["User"], "resourceTypes": ["Resource"] } },
      "vault::kv::write": { "appliesTo": { "principalTypes": ["User"], "resourceTypes": ["Resource"] } },
      "vault::pki::issue":{ "appliesTo": { "principalTypes": ["User"], "resourceTypes": ["Resource"] } },
      "argocd::sync":     { "appliesTo": { "principalTypes": ["User"], "resourceTypes": ["Resource"] } },
      "argocd::rollback": { "appliesTo": { "principalTypes": ["User"], "resourceTypes": ["Resource"] } },
      "trivy::scan":      { "appliesTo": { "principalTypes": ["User"], "resourceTypes": ["Resource"] } },
      "grafana::dashboards::read": { "appliesTo": { "principalTypes": ["User"], "resourceTypes": ["Resource"] } }
    },
    "commonTypes": {
      "ContextRecord": {
        "type": "Record",
        "attributes": {
          "human_approval":   { "type": "Boolean", "required": false },
          "incident_round":   { "type": "Long",    "required": false },
          "incident_phase":   { "type": "String",  "required": false },
          "verdict_status":   { "type": "String",  "required": false },
          "tone_review_passed": { "type": "Boolean", "required": false }
        }
      }
    }
  }
}
```

The schema deliberately mirrors the attribute set the existing `tools.cedar` already uses so `cedar validate` accepts both files against the same schema. A new lint rule under `scripts/validate-schema-coverage.sh` confirms every action referenced by `tools.cedar` is declared in the schema (catches drift the other way).

### 3.5 Template file (`agent-class.cedar.tmpl`)

Handlebars-style template that shapes generator output. The template is deliberately small so reviewers can audit it end-to-end:

```handlebars
// Generated from {{recipe_id}}.md (upstream: {{upstream}}) on {{generated_at}}.
// DO NOT EDIT BY HAND. Override in tools.cedar.

{{#each tools}}
permit (
    principal in [
        {{#each suggested_agent_classes}}
        User::"{{this}}"{{#unless @last}},{{/unless}}
        {{/each}}
    ],
    action == Action::"{{name}}",
    resource
){{#if requires_guard}} when {
    context.human_approval == true
}{{/if}};

{{/each}}
```

`requires_guard` is computed by the generator (`true` for `write`, `exec`, `mutate-with-undo` modes unless `safe_writes: true`).

### 3.6 Pi adapter

`tools/pi-compat-layer/cedar-emit.sh` reads `tools-generated.cedar` plus a Pi-host overlay (`tools/pi-compat-layer/pi-overlay.yaml`) and emits `tools/pi-compat-layer/dist/pi-tools.json` — the shape Pi's `AGENTS.md` / `SYSTEM.md` expects when granting tool access. Pi has no native Cedar evaluator (Pi intentionally avoids built-in MCP per the project's design philosophy), so the adapter renders Cedar `permit` rules as Pi tool allowlists, dropping `forbid` rules that have no Pi-side equivalent. The script logs every rule it drops so reviewers can spot enforcement gaps.

Output shape (consumed by Pi at agent-load time):

```jsonc
{
  "generated_from": "tools-generated.cedar",
  "generator_version": "f1-v0.1",
  "agents": {
    "collector-security": {
      "allow_tools": [
        "vault::kv::read",
        "falco::events",
        "k8s::audit-log"
      ],
      "deny_tools": [],
      "notes": "Generated. Pi has no context-aware guards; write-mode tools are filtered out and require human-operator handoff."
    },
    "human-operator": {
      "allow_tools": ["*"],
      "deny_tools": [],
      "notes": "Pi grants human-operator full surface; opsbench gateway (F2) re-imposes guards at request time."
    }
  }
}
```

Because Pi cannot evaluate `context.human_approval` at tool-dispatch time, write-mode actions get pushed into the `human-operator` allowlist (which Pi already prompts the operator about per the `AGENTS.md` convention). The F2 gateway re-evaluates the full Cedar policy server-side, so the security guarantee survives the Pi-side downshift — Pi is the UX layer, the gateway is the enforcement layer.

### 3.8 CI workflow shape

A new file, `.github/workflows/validate-cedar.yml`, runs on `pull_request` and `push` to `main`:

```yaml
name: validate-cedar
on:
  pull_request:
    paths:
      - 'packages/team-incident-response/mcp-recipes/**'
      - 'packages/team-incident-response/policies/cedar/**'
      - 'scripts/generate-cedar-policy.sh'
      - 'scripts/install-cedar-tools.sh'
      - 'tools/pi-compat-layer/**'
      - '.github/workflows/validate-cedar.yml'
  push:
    branches: [main]
jobs:
  generate-and-validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: install cedar tooling
        run: ./scripts/install-cedar-tools.sh
      - name: download previous snapshot
        uses: actions/download-artifact@v4
        with: { name: tools-generated-snapshot, path: .snapshots/ }
        continue-on-error: true
      - name: generate
        run: ./scripts/generate-cedar-policy.sh
      - name: validate tools.cedar
        run: cedar validate --schema packages/team-incident-response/policies/cedar/opsbench-schema.cedarschema.json --policies packages/team-incident-response/policies/cedar/tools.cedar
      - name: validate tools-generated.cedar
        run: cedar validate --schema packages/team-incident-response/policies/cedar/opsbench-schema.cedarschema.json --policies packages/team-incident-response/policies/cedar/tools-generated.cedar
      - name: diff vs snapshot
        if: github.event_name == 'pull_request'
        run: scripts/cedar-diff-snapshot.sh > diff-summary.md
      - name: comment on PR
        if: github.event_name == 'pull_request'
        uses: marocchino/sticky-pull-request-comment@v2
        with: { path: diff-summary.md }
      - name: upload snapshot
        if: github.ref == 'refs/heads/main'
        uses: actions/upload-artifact@v4
        with:
          name: tools-generated-snapshot
          path: packages/team-incident-response/policies/cedar/tools-generated.cedar
      - name: pi adapter
        run: ./tools/pi-compat-layer/cedar-emit.sh
      - name: validate pi-tools.json
        run: jq -e . tools/pi-compat-layer/dist/pi-tools.json
```

The job is fast (<2 min including tooling install) and runs in the same matrix slot as `lint:md` to avoid CI fan-out cost.

## 4. Cedar policy posture

F1 itself does not evaluate Cedar at runtime; the policy posture here is about what the F1 *artifacts* expose.

| Concern | F1 posture |
| ------- | ---------- |
| Default decision | Deny. Generator never emits a blanket `permit (principal, action, resource);` rule. |
| Read tools | Emit `permit` rules scoped to `suggested_agent_classes`, no `when` guard required. |
| Write/exec/mutate-with-undo tools | Emit `permit` rules guarded by `context.human_approval == true` *unless* the recipe sets `safe_writes: true` (e.g., scratch namespaces). |
| Override priority | Hand-written `tools.cedar` is loaded second so its rules take precedence in cases where `cedar` evaluator order matters. The F2 gateway documents the merge semantics. |
| Schema | `policies/cedar/opsbench-schema.cedarschema.json` is hand-maintained and lists every action namespace (`vault::*`, `argocd::*`, `k8s::*`, etc.). New recipes that reference unknown namespaces fail `cedar validate`. |
| Audit | `validate-cedar` CI job uploads the generated file as a build artifact so reviewers can diff between PRs. |
| Generator authority | The generator runs unprivileged. It cannot read or write keys, secrets, or the F3 Ed25519 private key (F3 enforces mode 0600 on `~/.config/opsbench/keys/gateway.key`). |
| Generated-file tampering | `.gitignore` + the CI job re-running the generator means a hand-edit to `tools-generated.cedar` either gets clobbered locally or fails the CI diff snapshot. |
| Recipe-level supply-chain risk | A malicious recipe could declare a permissive `tools:` block. Mitigation: every recipe goes through PR review; the generator also surfaces a diff summary (`+N permits, -M permits`) in the PR comment via `validate-cedar`. |

## 5. Acceptance criteria

1. `scripts/generate-cedar-policy.sh` exists, is executable (`chmod +x`), is bash 4.x+ compatible, and exits 0 on a clean checkout once `scripts/install-cedar-tools.sh` has run.
2. Running the generator with **zero** recipes containing a `tools:` block produces an empty (header-only) `tools-generated.cedar` and exits 0 with a warning summary to stderr.
3. Running the generator against the F0-shipped recipes (~30) produces a `tools-generated.cedar` file that passes `cedar validate --schema policies/cedar/opsbench-schema.cedarschema.json --policies tools-generated.cedar`.
4. Running the generator twice in a row on an unchanged repo produces byte-identical `tools-generated.cedar` output (the test uses `cmp -s`).
5. The `tools-generated.cedar` file is listed in `.gitignore` and is not committed.
6. A new CI job `validate-cedar` runs on every PR. It:
   - Installs `cedar` and `cedar-for-agents` via `scripts/install-cedar-tools.sh`.
   - Runs the generator.
   - Validates both `tools.cedar` (existing) and `tools-generated.cedar` (new).
   - Diffs the generated file against the previous run's snapshot stored as a workflow artifact; if the diff is non-trivial, posts a summary comment on the PR (`+ N permit rules`, `- M permit rules`, list of recipe IDs that changed).
   - Fails when:
     a. The generator exits non-zero.
     b. `cedar validate` rejects either file.
     c. A recipe references an unknown agent class (warning today, hard fail once F4 ships team-level agent classes).
7. The 16 pre-existing recipes under `packages/team-incident-response/mcp-recipes/` each gain a valid `tools:` frontmatter block in a separate PR (`feat/f1-recipe-tools-frontmatter`). The 16-file diff lands in one PR; each recipe's block is reviewed against the upstream MCP server's actual tool surface.
8. `tools/pi-compat-layer/cedar-emit.sh` exists, runs against `tools-generated.cedar`, and produces a syntactically valid Pi `pi-tools.json` manifest (validated with `jq` against a schema in `tools/pi-compat-layer/pi-tools.schema.json`).
9. `pnpm install` (top-level) runs the generator as a `prepare` step. The doctor script (`scripts/doctor.sh`) gains a check that confirms the generator was run since the last recipe change (compares mtimes).
10. The PR that adds the generator (PR A) does not touch any existing recipe; the PR that backfills `tools:` blocks (PR B) does not touch the generator. The two PRs are independently revertable per the F-series "standalone PRs" cross-cutting principle.
11. `tools.cedar` is untouched in F1. A follow-up issue is filed for the F4 cleanup pass.
12. A new doc, `packages/team-incident-response/policies/cedar/README.md`, explains the merge semantics (hand > generated) and tells reviewers how to add a new recipe's tools without bloating the override file.

## 6. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
| ---- | ---------- | ------ | ---------- |
| `cedar-policy/cedar-for-agents` is immature (30★, last activity 2026-05-26); breaking schema or CLI changes | Medium | High — breaks generator | Pin to a specific git SHA in `scripts/install-cedar-tools.sh`. Vendor the binary as a CI cache key. Add a smoke test (`scripts/test-cedar-tools.sh`) that exercises the pinned CLI on a fixture so we detect drift on upgrade. |
| `cedar-for-agents` may not expose the `convert` subcommand exactly as documented above | High | Medium | The exact CLI surface is verified during implementation; if it differs, the generator wraps a Rust subcommand exposed by the crate's `main.rs`. Worst case: a small Rust shim under `tools/cedar-for-agents-shim/` that calls the library directly. |
| Generator wall-clock time grows with recipe count | Low | Low | ~38 recipes × <100ms each = <4s. Even with F4 adding 5 teams × ~10 recipes each, total stays under 10s. Re-evaluate at 200+ recipes. |

## 7. Open questions for reviewer

1. **Frontmatter vs sibling `tools.yaml`.** The design lets recipes use either. Is allowing both worth the parser complexity, or should F1 mandate frontmatter only and defer `tools.yaml` to a future opt-in? 2. **Generator pinning strategy.** Pin `cedar-for-agents` to a git SHA (deterministic but stale) or to a semver tag once one exists (closer to upstream)? Bias: SHA pin until the project crosses a v0.1 tag; switch to tag-pinning when it does.
3. **CI snapshot storage.** Store the previous `tools-generated.cedar` snapshot as a GitHub Actions artifact (cheap, ephemeral, loses history beyond 90 days) or commit it under `.snapshots/`? Bias: artifact + a quarterly snapshot committed under `docs/superpowers/snapshots/<date>-tools-generated.cedar` for long-term archaeology.
