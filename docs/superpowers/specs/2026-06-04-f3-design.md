## 1. Purpose

F0–F2 ship an auditable choke-point and a JSON-Lines custody log that records every tool call with hashes of args and responses. That gets opsbench to "tamper-evident when the log file is intact" — but a custody-log line is just text on disk: a sophisticated attacker who reaches the host can rewrite history with `sed` and nobody downstream of the file can tell. F3 closes that hole by signing every custody entry with an Ed25519 key bound to the gateway process at install time, and by shipping a verifier that re-evaluates each line offline against a published public key. The gateway becomes the only entity that can produce valid receipts; an investigator with the public key can detect any tampering, replay, or splice without trusting the host where the log lives. The signing format and key bootstrap are adopted from `scopeblind-gateway` (the closest existing analog the catalog surfaced); we do not vendor scopeblind — we adopt the shape and fold it into opsbench-gateway and the legacy `post-tool-use.sh` so both surfaces converge on `custody-log.v2.json`. This phase ships only the signing path, the key-management bootstrap, the verifier CLI, and the schema/version bump; gateway architecture from F2 and team-package work from F4 are unchanged.

## 2. Scope

### In-scope

- **Schema bump.** New file `packages/team-incident-response/schemas/receipt.v1.json` (draft 2020-12, the "F3 receipt envelope") plus `packages/opsbench-gateway/schemas/custody-log.v2.json` (the same envelope inlined into the gateway's writer schema). The v2 custody-log schema is additive on top of F2's v1: it adds `schema_version: "v2"`, `signer_pubkey`, `signature`, `signed_at`, `signed_payload_canonical_sha256`. Every v1 field stays present, in the same JSON-Pointer location, with the same type.
- **Ed25519 key bootstrap in `scripts/install.sh`.** New stage that runs after the existing per-team install: detects whether `~/.config/opsbench/keys/gateway.key` exists; if not, generates an Ed25519 keypair (private key + public key) with `openssl genpkey -algorithm Ed25519`, persists the private key at `~/.config/opsbench/keys/gateway.key` (mode 0600, owner-only), the public key at `~/.config/opsbench/keys/gateway.pub` (mode 0644), and a UUIDv4 `signer_id` at `~/.config/opsbench/keys/gateway.id`. The user is shown the public-key fingerprint (`SHA-256` of the SubjectPublicKeyInfo DER) and a one-line copy-paste for sharing.
- **Receipt-signing path in opsbench-gateway.** New Go package `packages/opsbench-gateway/internal/signing/`. The custody writer (F2 § 3.4) calls `signing.Sign(ctx, entry)` before the JSON-Lines line hits disk. The signer loads the private key once at startup; rotation is a graceful restart. Configuration knob in `gateway.yaml`: `custody.schema_version: "v2"` (default in F3 onward) or `"v1"` (opt-out path for users with strict performance budgets, documented as unsafe).
- **Receipt-emission path in `post-tool-use.sh`.** The legacy hook also gains signing. New helper `packages/team-incident-response/hooks/lib/sign-receipt.sh` invokes `openssl pkeyutl -sign` against the canonicalised entry and appends both the SHA-256 line (back-compat) and a sibling `receipts.jsonl` line in the v2 schema. Hosts that have *not* installed an opsbench-gateway still produce signed receipts through the hook path — this is the "Pi or Claude Code without gateway" fallback.
- **Verifier CLI.** Two surfaces, identical semantics:
  1. **Standalone Bash:** `scripts/verify-receipts.sh <path/to/custody.log> [--pubkey <path>] [--strict]`. Pure `openssl` + `jq`; no Go toolchain required. Ships in the opsbench installer.
  2. **Gateway subcommand:** `opsbench-gateway custody verify-signatures <path> [--pubkey <path>] [--allow-rotated]`. Same exit codes, richer output (line-numbered failures, summary stats).
- **Receipt-verify hook.** `post-tool-use.sh` gains an optional pre-write verification step: if the *incoming* event has a parent receipt (from a prior agent in the chain — e.g. an orchestrator routing to a sub-collector), the hook verifies the parent receipt's signature before writing its own line. This makes the custody log a *chain*, not just a list. Receipts carry an optional `parent_receipt_sha256` field; F3 emits and verifies this for nested calls.
- **Optional offsite mirror.** `gateway.yaml#custody.mirror` adds an S3 / Azure Blob / GCS sink. Each receipt is mirrored (append-only) within `gateway.yaml#custody.mirror.flush_interval` (default 5s). The mirror schema is identical to the on-disk format; the bucket/container is treated as tamper-evident-when-immutable-policy-enabled (object lock recommended). This is opt-in.
- **Key rotation runbook.** `packages/opsbench-gateway/UPSTREAM-SYNC.md` already documents toolhive sync; F3 adds `scripts/rotate-gateway-key.sh` and a doc `packages/team-incident-response/docs/key-rotation-runbook.md` that lays out: generate-new → restart gateway with both keys → drain in-flight → retire-old → publish new fingerprint. Old receipts stay verifiable because the public-key fingerprint is in every receipt; verifiers maintain a fingerprint→pubkey map.
- **CI.** `.github/workflows/receipts-validate.yml` runs the verifier against a fixture custody.log on every PR that touches `packages/opsbench-gateway/**` or `packages/team-incident-response/hooks/**` or the schemas. Also runs a tamper-detection test: mutate a byte in the fixture, expect the verifier to fail with `signature-mismatch` exit code.
- **Pi-first config docs.** The `~/.pi/agent/AGENTS.md` snippet in each F2-rewritten recipe gains a note: "F3-signed receipts active when gateway is configured; private key at `~/.config/opsbench/keys/gateway.key`." CLI-Anything wrapper bundles for hosts that lack native gateway routing get an inline `sign-receipt.sh` step so they too participate in the signed chain.

### Out-of-scope

- **Asymmetric algorithms other than Ed25519.** No ECDSA P-256, no RSA, no Ed448. F3 is opinionated. F6 may reopen this if FIPS-compliance becomes a hard requirement.
- **Hardware-backed keys (HSM / YubiKey / TPM / Secure Enclave).** The private key lives in a file. F6 evaluates HSM integration once we have real users asking for it.
- **PKI / X.509 / certificate chains.** Receipts carry raw public keys (raw32 / SPKI DER), not certs. There is no CA hierarchy. Users distribute pubkeys out-of-band (Git commit, internal wiki, sealed-secret in a config repo). The `signer_id` UUID is the join key, not a Subject DN.
- **Encryption of custody.log.** Receipts are signed, not encrypted. The plaintext payload is still legible (which is the point — auditors need to read it). Encryption at rest is the operator's concern (LUKS / dm-crypt / cloud KMS at the volume level).
- **Per-tool key segregation.** One key per gateway instance. Users who want per-team segregation run multiple gateway instances (the single-host / single-tenant assumption from F2 § 2 still holds).
- **Distributed signing / threshold signatures.** No FROST, no multi-sig. F3 is "this host produced this receipt, here is the proof". Cross-host correlation is F6 territory.
- **Time-stamping authority (RFC 3161).** Trusted timestamps are not in scope. Receipt `ts` is the gateway's `time.Now()`. Operators who need notarised timestamps can run a separate TSA pipeline against the rotated logs.
- **Schema-version >= 3.** F3 ships v2 only. Future bumps (v3, …) follow the same additive contract this spec establishes.
- **Retroactive signing of pre-F3 custody.log entries.** Existing v1 entries stay v1. The verifier knows the format and treats them as `unsigned: true`.

### 3.2 Canonical payload (what gets signed)

To make the signature reproducible across implementations (Go gateway, Bash hook, future Pi-native helper), F3 defines a canonical-JSON form:

1. Take the receipt object as a map.
2. Remove the four signature-envelope fields (`signature`, `signed_at`, `signed_payload_canonical_sha256`, and `signer_pubkey_fingerprint` — keep `signer_pubkey`).
3. Sort keys lexicographically at every depth.
4. Render with no insignificant whitespace, `\n` line endings, UTF-8 encoding, integers as decimal (no `+`, no leading zeros), strings JSON-escaped per RFC 8259.
5. Compute `signed_payload_canonical_sha256 = SHA-256(canonical_bytes)` for the fast pre-check field.
6. Sign `canonical_bytes` with Ed25519 → `signature`.
7. Set `signed_at = time.Now().UTC().Format(RFC3339Nano)`.
8. Set `signer_pubkey_fingerprint = "sha256:" + hex(SHA-256(SPKI_DER(public_key)))`.
9. Reassemble the full receipt with all envelope fields populated; write as a single JSON-Lines line.

The canonical form is identical to JCS (RFC 8785) restricted to the keys present in the receipt minus the four envelope fields. We do not require an external JCS library — the deterministic key sort + JSON.Marshal in Go and `jq -cS` in Bash produce byte-identical output for the JSON shapes we use (no floats, no NaN, no Unicode normalisation edge cases because all string fields are ASCII-or-base64 by construction).

### 3.4 Per-receipt lifecycle inside the gateway

```text
1. router receives MCP envelope, parses tool + args
2. cedar evaluator returns decision + matched_rule_ids
3. custody writer assembles partial entry (F2 v1 fields):
       ts, gateway_id, agent_class, upstream_id, tool, args_sha256, response_sha256,
       decision, deny_reason, policy_files, matched_rule_ids, latency_ms,
       redacted_args, redacted_response_preview, context, error_class
4. signing module:
       a. injects signer_id (loaded from ~/.config/opsbench/keys/gateway.id)
       b. injects signer_pubkey (raw32 hex)
       c. computes canonical_bytes per § 3.2 steps 1-4
       d. signed_payload_canonical_sha256 = sha256(canonical_bytes)
       e. signature = ed25519.Sign(private_key, canonical_bytes)
       f. signed_at = time.Now().UTC()
       g. signer_pubkey_fingerprint = "sha256:" + hex(sha256(SPKI_DER(public_key)))
5. writer marshals the final object with json.Marshal (deterministic for our fields)
6. fsync (per gateway.yaml#custody.fsync_each_line)
7. if mirror enabled: enqueue to the mirror channel; flush every flush_interval
```

The signing step adds ~50µs per call on a modern CPU (Ed25519 sign is fast) plus the canonical-JSON SHA-256, which dominates at ~10µs for typical entries. Total budget impact: well under 100µs per call, inside F2's <100ms p95 round-trip ceiling.

### 3.5 Per-receipt lifecycle inside `post-tool-use.sh` (no-gateway path)

```text
1. PostToolUse fires; stdin JSON event arrives (existing F0 contract)
2. existing logic computes sha256 of output_path and appends pipe-delimited
       custody.log line (back-compat, unchanged)
3. NEW: lib/sign-receipt.sh is sourced
4. it builds a JSON object matching schemas/receipt.v1.json with whatever
       fields are available (gateway_id="", since there is no gateway —
       fallback is signer_id-only chain-of-custody)
5. it serialises via `jq -cS . | tee /tmp/opsbench-canonical.json`
6. it runs `openssl pkeyutl -sign -inkey ~/.config/opsbench/keys/gateway.key \
       -rawin -in /tmp/opsbench-canonical.json -out /tmp/opsbench-sig.bin`
7. converts sig to hex via `xxd -p -c 256`
8. re-assembles the final receipt with the envelope fields populated
9. appends one line to $INCIDENT_DIR/receipts.jsonl (parallel to custody.log)
```

The hook continues to exit 0 on any failure — signing is best-effort in the legacy path, because the hook MUST NOT block tool execution. If `openssl` is missing, the hook logs a one-line warning to `/tmp/k8s-incident-post-tool-use.jsonl` and skips the receipt. This matters because the gateway path is the strong guarantee; the hook path is the "still better than v1" fallback. The CI receipts-validate workflow only fails for the gateway path.

### 3.6 Key bootstrap in `scripts/install.sh`

A new stage runs after the existing prefix preparation, before per-team copy. Sketch:

```bash
ensure_signing_keys() {
    local keys_dir="${HOME}/.config/opsbench/keys"
    local priv="${keys_dir}/gateway.key"
    local pub="${keys_dir}/gateway.pub"
    local sid="${keys_dir}/gateway.id"

    mkdir -p "${keys_dir}"
    chmod 0700 "${keys_dir}"

    if [[ -f "${priv}" && -f "${pub}" && -f "${sid}" ]]; then
        ok "signing keys present at ${keys_dir}"
        return 0
    fi

    info "generating Ed25519 signing key at ${priv}"
    require_cmd openssl
    openssl genpkey -algorithm Ed25519 -out "${priv}" 2>/dev/null
    chmod 0600 "${priv}"
    openssl pkey -in "${priv}" -pubout -out "${pub}"
    chmod 0644 "${pub}"

    # signer_id is a UUIDv4
    if command -v uuidgen >/dev/null 2>&1; then
        uuidgen | tr 'A-Z' 'a-z' > "${sid}"
    else
        # python3 fallback shipped with every supported platform
        python3 -c 'import uuid; print(uuid.uuid4())' > "${sid}"
    fi
    chmod 0644 "${sid}"

    local fp
    fp="$(openssl pkey -in "${priv}" -pubout -outform DER \
          | openssl dgst -sha256 -binary | xxd -p -c 256)"
    ok "signer_id=$(cat "${sid}")"
    ok "pubkey fingerprint=sha256:${fp}"
    info "share the fingerprint with auditors out-of-band; the private key never leaves this host"
}
```

The bootstrap is idempotent. Re-running the installer is safe. `--dry-run` prints the planned actions and does not touch the filesystem. If the user already has a key (e.g. from a previous install or a manual rotation), the installer leaves it alone and emits the existing fingerprint.

### 3.7 Verifier semantics (Bash + Go subcommand, same exit codes)

Inputs:

- A custody.log path (JSON-Lines, v1 or v2 entries intermixed).
- Optional `--pubkey <path>` (if omitted, the verifier reads `~/.config/opsbench/keys/gateway.pub`).
- Optional `--pubkey-map <path>` for rotated-key environments (JSON: `{ "<signer_id>": "<pubkey_hex>" }`).
- Optional `--strict` (gateway-path-only — also fails on any v1 unsigned line).

Per-line algorithm:

```text
for each line in input:
    parse JSON → entry
    if entry.schema_version == "v1":
        if --strict: fail; else: count as "unsigned, skipped"
    if entry.schema_version == "v2":
        resolve pubkey:
            if --pubkey-map and entry.signer_id in map: use mapped key
            else: use --pubkey or the default ~/.config/opsbench/keys/gateway.pub
        rebuild canonical_bytes per § 3.2 steps 1-4
        compute sha256(canonical_bytes)
        if sha256 != entry.signed_payload_canonical_sha256:
            fail with line+reason "canonical-hash-mismatch"
        verify ed25519(pubkey, canonical_bytes, entry.signature)
        if verify fails: fail with line+reason "signature-mismatch"
        if entry.parent_receipt_sha256 is non-null:
            require the parent receipt (previous lines or another file) to exist
            and its sha256 to match; fail with "parent-chain-broken" if not
print summary: total / signed / unsigned / failed
exit 0 if zero failures, else exit 2
```

Exit codes (shared between Bash and Go):

| Code | Meaning |
|------|---------|
| 0 | All v2 lines verified; v1 lines tolerated unless `--strict` |
| 2 | One or more lines failed verification |
| 3 | I/O error or missing pubkey |
| 4 | Malformed input (not JSON-Lines) |
| 5 | `--strict` failure on v1 entries |

### 3.8 Optional offsite mirror

`gateway.yaml#custody.mirror`:

```yaml
custody:
  path: "~/.local/share/opsbench/custody.log"
  rotate_at_bytes: 134217728
  keep_rotated: 8
  fsync_each_line: true
  schema_version: "v2"

  mirror:
    enabled: false
    backend: "s3"             # one of: s3 | azure-blob | gcs | none
    bucket: "opsbench-evidence"
    prefix: "gateway-${gateway_id}/"
    region: "us-east-1"       # backend-specific
    flush_interval: 5s
    object_lock: true         # require bucket-level immutability; refuse to start if not enabled
    credentials_from: "env"   # one of: env | iam | imds | sa
```

The mirror writer is a single goroutine reading from a bounded channel; the main writer never blocks on the network. On flush failure the receipts stay queued; if the channel saturates the gateway logs a warning and applies backpressure to the custody writer (which causes new MCP calls to fail with `error_class: "mirror-saturated"` — surfaced in the agent's response so the operator notices). The mirror layout is one object per flush window, named `<unix-ms>-<seq>.jsonl`; objects are append-only by construction.

Object lock + WORM-style immutability is the operator's responsibility — F3 just refuses to start if the backend says lock is disabled (a single API call per bucket at startup).

### 3.9 Chain-of-receipts (`parent_receipt_sha256`)

When an agent's tool call triggers a sub-call (an orchestrator routing to a sub-collector, a multi-agent skill fanning out), the gateway forwards the parent receipt's SHA-256 as a request header (`X-Opsbench-Parent-Receipt-Sha256`). The downstream gateway invocation reads that header and populates `parent_receipt_sha256` in the child receipt. The verifier walks the chain at verify time and surfaces any broken links.

This is opt-in per upstream (`gateway.yaml#upstreams[].propagate_parent_receipt: true`) and defaults to true for the team-incident-response gateway profile.

### 3.11 Backwards-compatibility matrix

| Reader | Encounters v1 entry | Encounters v2 entry |
|--------|---------------------|---------------------|
| F2 gateway custody writer | Writes v1 (its native schema) | N/A (won't be running once F3 ships) |
| F3 gateway custody writer | Reads it (e.g. on log rotation roll-up); preserves it | Writes it |
| F2 verifier (none existed) | N/A | N/A |
| F3 Bash verifier | "unsigned, skipped" unless `--strict` | Verifies signature |
| F3 Go verifier (gateway subcommand) | "unsigned, skipped" unless `--strict` | Verifies signature |
| External JSON tool | Always reads — every v1 field present in v2 | Reads ignored envelope fields |

The schema-version bump is *additive*, *position-stable*, and *forward-extension-only*. No field is renamed, no field's type is narrowed, no field's semantic changes.

### 3.13 Pi-first + other-host posture

- **Pi (primary).** Pi has no native MCP, so the receipt path is *always* via opsbench-gateway when the user goes through the recipe lead block (F2 § 3.8). The Pi `AGENTS.md` snippet is unchanged — the gateway URL still points at `http://localhost:8765/mcp/<id>`. Receipt signing is invisible to Pi.
- **Claude Code (secondary).** Two paths: (a) via gateway, identical to Pi; (b) via `PostToolUse` hook in `.claude/`, where the legacy `post-tool-use.sh` is what writes the receipt. The legacy path produces the same v2 envelope, just without `upstream_id` and with `gateway_id: ""`.
- **Codex CLI, Cursor, Gemini, OpenCode, GitHub Copilot.** The recipes already point each host at the gateway via `tools/<host>-compat-layer/` (F5 work). The signing path is identical because it runs inside the gateway, not inside the host.
- **CLI-Anything wrapped hosts.** Hosts that lack first-class MCP support ship a CLI-Anything wrapper recipe (parent roadmap § "Pi-first multi-host"). The wrapper script reads the `OPSBENCH_GATEWAY_URL` env var and proxies the call through the gateway; the receipt is produced there. For hosts without even CLI-Anything plumbing, the documentation directs the user to the `post-tool-use.sh` fallback and notes the lost gateway-side fields.

## 4. Cedar policy posture

F3 touches Cedar in three concrete ways even though the evaluation engine itself does not change:

| Concern | F3 posture |
|---------|-----------|
| Receipt-signing as a Cedar-eligible event | The signing step is policy-free — every allowed *or* denied decision still produces a receipt. There is no "don't sign this" Cedar rule, by design (the audit log is non-negotiable per parent roadmap § "Cross-cutting principles"). |
| Policy provenance in the signed payload | `policy_files` and `matched_rule_ids` are inside the canonical payload, so a verifier can prove offline which rule produced the decision. F2 already populated these for v1; F3 ensures they are signed. |
| Key-management actions are Cedar-gated | The `scripts/rotate-gateway-key.sh` script writes to `~/.config/opsbench/keys/`. The corresponding `Action::"opsbench-gateway::keys::rotate"` is added to `opsbench-schema.cedarschema.json`. By default `tools.cedar` carries: `forbid(principal, action == Action::"opsbench-gateway::keys::rotate", resource) when { !context.human_approval };` Rotation thus requires `human_approval == true` plus the HMAC-signed header from F2 § 4. |
| Verifier never needs Cedar | The verifier is *offline*. It does not call Cedar. It only checks signatures. This is intentional — auditing must be cheap and run anywhere, even on a laptop with no Cedar runtime. |
| Default-deny still owns the runtime | F2's "schema closes the world" remains the runtime fence. F3 just makes the *record* of those denials cryptographically tamper-evident. Deny entries are signed identically to allow entries. |
| `signer_id` in Cedar context | The gateway's `signer_id` is exposed in `context.signer_id` so policy authors can write rules like `forbid(principal, action, resource) when { context.signer_id != "<expected-uuid>" };` for environments that need to assert "only this key may operate this profile". This is an opt-in pattern, documented in `key-rotation-runbook.md`. |
| Hot-reload of key vs policy | Cedar policies hot-reload on `SIGHUP` (F2 § 4). Key rotation requires a graceful restart (the signing module loads the key once at startup, then never reads from disk). The runbook spells out the order: rotate key → restart gateway → restart hooks. |
| Forward-compat with F4 team-specific keys | F4 team packages may need per-team signing identities. The schema already supports this — each team's gateway profile carries its own `signer_id` and `signer_pubkey`; the verifier's `--pubkey-map` already handles N keys. No schema change needed in F4. |

## 5. Acceptance criteria

1. **Schemas land at the documented paths.**
   - `packages/team-incident-response/schemas/receipt.v1.json` exists and validates against draft 2020-12 (`ajv compile` exit 0).
   - `packages/opsbench-gateway/schemas/custody-log.v2.json` exists, references `receipt.v1.json` via `$ref`, validates.
   - Existing `custody-log.v1.json` is unchanged.
2. **Ed25519 key bootstrap in `scripts/install.sh`.**
   - First fresh install generates `~/.config/opsbench/keys/{gateway.key,gateway.pub,gateway.id}` with modes `0600/0644/0644` and a `0700` parent dir.
   - Re-running the installer is a no-op (idempotent) and prints the existing fingerprint.
   - `--dry-run` prints the planned actions and creates no files.
   - The installer prints the SHA-256 pubkey fingerprint exactly once at install completion.
3. **Gateway signing path.**
   - `packages/opsbench-gateway/internal/signing/` ships `ed25519.go`, `canonical.go`, and a unit test covering the canonical-JSON contract (key sort, no insignificant whitespace, escape handling).
   - A Go round-trip test signs an entry, mutates a single byte of the canonical form, and asserts verification fails.
   - The custody writer emits v2 entries when `gateway.yaml#custody.schema_version: "v2"` (default in F3 builds).
   - With `custody.schema_version: "v1"`, the writer still emits v1; the operator must opt out explicitly.
   - Signing adds <200µs per call at p95 in the integration-test workload (assert in `gateway-integration.yml`).
4. **Hook signing path.**
   - `packages/team-incident-response/hooks/lib/sign-receipt.sh` exists and is sourced by `post-tool-use.sh`.
   - With `openssl` and the private key present, every PostToolUse event produces a line in `$INCIDENT_DIR/receipts.jsonl`.
   - With `openssl` missing, the hook still exits 0 and logs a warning; no receipt is produced.
   - The legacy pipe-delimited custody.log line continues to be written for back-compat.
5. **Verifier — Bash.**
   - `scripts/verify-receipts.sh <fixture-v2.jsonl>` exits 0 and prints `signed=N unsigned=0 failed=0`.
   - Mutating one byte of a v2 entry yields exit 2 with `signature-mismatch` and a line number.
   - `scripts/verify-receipts.sh <fixture-mixed.jsonl>` (v1+v2) exits 0 by default; with `--strict` exits 5.
   - `scripts/verify-receipts.sh --pubkey-map <map.json> <fixture-rotated.jsonl>` verifies entries signed by either key in the map.
6. **Verifier — gateway subcommand.**
   - `opsbench-gateway custody verify-signatures <path>` matches the Bash semantics exit-code for exit-code.
   - `opsbench-gateway custody verify-signatures --allow-rotated <path>` accepts the `--pubkey-map` rotated set.
7. **Chain-of-receipts.**
   - Two consecutive gateway calls with `propagate_parent_receipt: true` produce receipts where the second's `parent_receipt_sha256` matches the SHA-256 of the first's canonical form.
   - The verifier flags a broken chain (synthetic test that drops the middle line) with `parent-chain-broken`.
8. **Offsite mirror (when enabled).**
   - With `gateway.yaml#custody.mirror.enabled: true` and a fixture S3-compatible backend (LocalStack in CI), receipts arrive in the bucket within `flush_interval + 1s`.
   - With `object_lock: true` and a bucket that does not have object lock enabled, the gateway refuses to start with a clear error.
   - Mirror saturation triggers backpressure (assert via a synthetic slow-mirror in CI).
9. **Key rotation.**
   - `scripts/rotate-gateway-key.sh` generates a new keypair, writes it alongside the old one (e.g. `gateway.key.next`), prints next steps, and is itself idempotent.
   - `key-rotation-runbook.md` exists and prescribes: generate next → publish new fingerprint → restart gateway → verify both keys in the verifier `--pubkey-map` → retire old (rename `.prev`) → optional `--strict` rerun.
10. **CI.**
    - `.github/workflows/receipts-validate.yml` runs on every PR touching `packages/opsbench-gateway/**`, `packages/team-incident-response/hooks/**`, or any receipt schema.
    - It signs a fixture, verifies it (expect exit 0), tampers with it, re-verifies (expect exit 2 + `signature-mismatch`), and verifies a v1-only fixture in `--strict` mode (expect exit 5).
    - The workflow runs both the Bash verifier and the gateway subcommand against the same fixtures and asserts identical exit codes.
11. **Documentation.**
    - `packages/opsbench-gateway/README.md` gains a "Signed receipts (F3)" section with the canonical-payload contract, the verifier examples, and a security note about key handling.
    - `packages/team-incident-response/docs/key-rotation-runbook.md` exists and covers rotation, lost-key recovery (declare bankruptcy + reseal log with a new key, sign a "rotation receipt"), and operator audit checklist.
    - `docs/integrations.md` gains a brief note under "Gateways" pointing to the F3 receipts entry.
12. **PR shape: at most 4 PRs, each independently revertable.**
    - PR 1: schemas + key bootstrap in `scripts/install.sh` + `scripts/verify-receipts.sh` + CI workflow.
    - PR 2: gateway `internal/signing/` package + writer integration + custody-log v2 wire.
    - PR 3: `post-tool-use.sh` + `lib/sign-receipt.sh` + receipts.jsonl path + tests.
    - PR 4: optional mirror + key rotation script + runbook + docs.
13. **Backwards compatibility.**
    - F2 v1 custody.log files remain readable by all F3 tooling (verifier counts them as `unsigned, skipped`).
    - A gateway running with `custody.schema_version: "v1"` continues to function exactly as in F2 (no signing, no envelope fields).
    - Recipes from F0/F2 require no changes for F3 — the gateway URL and config blocks are unchanged.

## 6. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| Private key compromise (host attacker reads `~/.config/opsbench/keys/gateway.key`) | Medium on shared hosts | Critical | Key is mode 0600 owner-only; parent dir 0700; documented in install output. The rotation runbook is the recovery path: rotate key, publish new fingerprint, sign a "rotation receipt" with the new key. Old receipts stay verifiable via the fingerprint embedded in each one. HSM integration is F6, not F3 — we are honest about the threat model. |
| Canonical-JSON divergence between Go and Bash implementations (a verified-in-Go receipt fails Bash verify or vice versa) | Medium | High | A shared fixture corpus (`packages/opsbench-gateway/internal/signing/testdata/`) is consumed by both implementations in CI. The fixture covers: Unicode strings (ASCII-only in this spec, but the test asserts non-ASCII rejected), nested objects, null fields, integer edge cases (0, 1<<31, 1<<53-1), empty arrays. CI fails if either implementation produces a different canonical SHA-256 from the fixture's expected value. |
| Performance regression from per-call signing | Low | Medium | Ed25519 sign is ~50µs on modern hardware; canonical JSON SHA-256 is ~10µs for typical entries. Combined budget impact <200µs at p95, well inside F2's <100ms p95 round-trip target. CI asserts the budget in `gateway-integration.yml`. Operators with extreme performance needs can opt out via `custody.schema_version: "v1"` (documented as unsafe). |
| Receipt schema growth pressure (F4 teams want more fields) | High | Medium | Schema is `additionalProperties: false`. New fields require a `receipt.v2.json` and `custody-log.v3.json`. Adding a field is a deliberate review process, not a stealth change. Forward-extension contract: new fields are appended at the end of the canonical sort order; old verifiers ignore them as long as they are not in `required`. |
| Verifier exit-code skew between Bash and Go | Medium | High | Single source of truth: exit codes documented in this spec (§ 3.7) and enforced by a shared CI matrix that runs both verifiers against identical fixtures and `diff`s their stdouts. |
| Lost private key (host wiped, no backup) | Low | High | Runbook covers declare-bankruptcy path: generate a new key, publish a "key-lost" notice, sign all *future* receipts with the new key. Past receipts remain verifiable as long as the old pubkey is still distributed; if not, they become "unsigned with provenance lost" — documented as the correct semantic outcome (the auditor cannot trust them, which is the honest answer). |
| Clock skew between signing time (`signed_at`) and Cedar context time | Low | Low | `signed_at` is the gateway's `time.Now()`. `ts` is the request arrival time. Both are RFC3339Nano UTC. They may differ by microseconds; that is acceptable. Receipts that need notarised time use an external TSA (out of scope, § 2). |
| Receipt fan-out to S3 mirror leaks args via a misconfigured redaction rule | Medium | Critical | Redaction (F2 § 3.4) runs *before* the canonical-JSON step, so redacted args are what get signed and mirrored. The CI receipts-validate job runs the `--grep-secrets` scan from F2 § 6 against the mirror fixture output as well as the local custody.log. |
| `openssl` is missing on a minimal host running only the legacy hook | Medium | Medium | The hook detects the absence, writes a warning to its audit log, and degrades to v1 unsigned behaviour. Documented in the hook header comment. The installer's preflight (already in `scripts/install.sh#REQUIRED_CMDS`) adds `openssl` so the gateway path always has it. |
| Object-lock requirement is too strict for users with non-immutable buckets | Medium | Low | The `object_lock: true` check is opt-in via `gateway.yaml#custody.mirror.object_lock`. Operators who accept the lower guarantee can set it to `false` and the gateway emits a startup warning explaining the reduced tamper-evidence guarantee. |
| Pi-side receipt fields are sparse because Pi has no gateway URL knowledge | Low | Medium | This is acceptable; Pi-side receipts (when running via the hook path on a Pi-CLI-Anything wrapper) carry `gateway_id: ""` and `upstream_id: ""` and a populated `agent_class`. The verifier treats this as valid v2. The documented downgrade is "weaker provenance, still tamper-evident". |
| F3 ships before F2 lands and orphans the schema | Low | Medium | The roadmap (§ "Dependency graph") locks F3 after F2. If pressure mounts to ship F3 first, the receipt envelope is still useful via the legacy hook path alone — but the canonical payload references gateway-only fields, so the hook path simply leaves those empty. No code change required; only an updated "Configuration — direct (advanced)" caveat. |

## 7. Open questions for reviewer

1. **Canonical JSON: roll-our-own deterministic Marshal vs. import a JCS library.** Default in this spec: roll our own, with a fixture-based equivalence test. The pure-stdlib path keeps the Go binary small and avoids a new transitive dependency in the Bash script. The downside is we shoulder the canonicalisation contract ourselves; if a future field requires non-ASCII strings or floats, the test surface grows. Reviewer's call.
2. **Mirror backend default.** Default in this spec: S3-compatible because LocalStack makes it easy to test in CI, and most users already have S3 credentials. The reviewer may prefer GCS or Azure Blob as the default for their own deployment (which we can express as a `gateway.yaml` template variation rather than a code change). The schema supports all three; the question is which we *document first*.
3. **Strict mode for the verifier on team-incident-response custody logs.** Default: strict is opt-in (verifier tolerates v1 lines). Reviewer may want the team-incident-response release notes to require `--strict` for incidents after a date `T`, so any v1 lines in a post-`T` log fail audit. This is a documentation choice, not a code change, but it is the most consequential of the three.
