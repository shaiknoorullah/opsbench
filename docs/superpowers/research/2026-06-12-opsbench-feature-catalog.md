# Opsbench Platform — Feature Catalog by Persona & Surface

## Executive Summary

This catalog consolidates the feature ideation of five persona agents — SRE/On-Call, DevOps/Platform Engineering, IT Operations, Support/Customer Operations, and Engineering Management/Executives — into a single deduplicated catalog of **116 features across seven themes**, each annotated with priority, delivery surfaces, key integrations, the personas it serves, and the rationale grounding it in market evidence.

Three strategic conclusions organize the catalog:

1. **Govern, don't compete.** The investigation layer is crowded (Datadog Bits, incident.io, Cleric, Rootly), the data moats are vendor-owned, and every system of record now ships an official MCP server — [Datadog](https://github.com/datadog-labs/mcp-server), [Grafana](https://github.com/grafana/mcp-grafana), [Kubernetes](https://github.com/containers/kubernetes-mcp-server), [Terraform](https://github.com/hashicorp/terraform-mcp-server), [Pulumi](https://www.pulumi.com/docs/ai/mcp-server/). The defensible position is the control plane above them: default-deny policy gateways, an actuation gatekeeper agents cannot write to, risk-tiered cross-surface approvals, signed evidence-grade audit, and earned-autonomy certificates backed by replay evals. No incumbent unifies observation with governed action.

2. **The cross-surface approval object is the product.** A hash-pinned payload with pre-minted idempotency key, TTL, named reviewer, and decision-ready diff — rendered identically across Slack, Teams, web, TUI, desktop tray, mobile Critical Alert, and voice DTMF, on one audit spine — is the correctness fix for the documented human-in-the-loop failure modes ([Truto](https://truto.one/blog/implementing-human-in-the-loop-approval-workflows-for-consequential-saas-api-actions/)): payload mutation between approval and execution, 29-second gateway timeouts, and rubber-stamp approval (93% approval rates measured by Anthropic; unverified — see evidence register).

3. **Trust is earned with evidence, not asserted with benchmarks.** Every market accuracy claim is self-reported (unverified: Komodor 95%, Traversal >90%, incident.io 80%) while ITBench-class results reportedly show SOTA agents resolving only 11.4–13.8% of real SRE scenarios (unverified). The platform's credibility moat is benchmark-honest, bring-your-own-incident evaluation feeding queryable, revocable autonomy certificates — the productized version of the read-only-first discipline the practitioner ecosystem already preaches ([k8slens guide](https://medium.com/k8slens/18-best-devops-mcp-servers-for-2026-the-definitive-guide-bfde04654a35): "start with read-only permissions, and scope access carefully before giving write access to production systems").

The document closes with a surface-by-persona matrix, an enterprise packaging sketch built on the market-consistent rule of gating governance rather than capability ([Datadog](https://www.datadoghq.com/pricing/), [incident.io](https://incident.io/blog/incident-management-pricing-comparison-2026), [Langfuse](https://langfuse.com/pricing) precedent), a three-phase delivery view, and an explicit register of unverified claims.

## How to Read This Catalog

**Persona codes:** SRE = SRE/On-Call · PLT = DevOps/Platform Engineering · ITO = IT Operations · SUP = Support/Customer Operations · EXE = Engineering Management/Executives. **All** = all five.

**Surface codes:** W = web · T = TUI · D = desktop tray · M = mobile · S = Slack/Teams (ChatOps) · V = voice · A = API.

**Priority:** P0 = launch-blocking table stakes or core differentiator; P1 = fast-follow; P2/P3 = horizon. Where personas proposed different priorities for a shared feature, the highest is kept.

**Evidence discipline:** Claims sourced to the practitioner reference corpus and the surfaces/enterprise research are cited inline. Claims originating only from ideation-agent research (vendor benchmarks, internal Google architectures, academic studies without links) are marked **(unverified)** and consolidated in the closing register.

## Theme 1: Incident Response & Investigation

The investigation loop — page, hypothesize, gather evidence, escalate, remediate, learn — is where the platform meets users first. The canonical reference implementation is Anthropic's [SRE Incident Responder cookbook](https://github.com/anthropics/claude-cookbooks/tree/main/managed_agents): PagerDuty webhook → agent reads logs → minimal diff → `request_approval` → merge only on `"approved"`. The platform productizes that spine and extends it with the customer-impact, SLA, and escalation context no investigator vendor ships.

## Theme 2: Agent Teams & Governance

This theme is the platform's center of gravity: the actuation control plane the agent cannot write to, deterministic policy outside the reasoning loop, and autonomy that is earned, certified, and revocable. The architectural premises: LLMs cannot enforce their own constraints (AWS AgentCore made gateway-intercepted Cedar policy the industry default at its March 2026 GA — unverified); Google SRE's Actus reasoning/actuation split is the production-proven reference (unverified); and the Anthropic cookbook's Iron Law — "Never call merge_pull_request unless request_approval returned 'approved'" — generalizes to every mutation.
