# MCP Recipe — kubernetes-mcp

Distro-agnostic Kubernetes/OpenShift MCP server maintained by the containers org. Called by the opsbench `incident-triage` and `recovery-executor` agent classes during cluster investigations — read-only by default (`--non-destructive`), with OpenTelemetry instrumentation so every tool call is traceable in the incident timeline. Pair with `kubernetes-cli-bridge-mcp` (kubectl/helm/argocd CLI wrap) when raw CLI behaviour is required; this recipe is the structured API surface.

```bash
docker pull ghcr.io/containers/kubernetes-mcp-server:latest
```

## Configuration — Pi (primary)

Pi (pi.dev) does not ship a built-in MCP runtime. Wrap `kubernetes-mcp-server` as a Pi-callable CLI via [`HKUDS/CLI-Anything`](https://github.com/HKUDS/CLI-Anything) and install the resulting skill from your fork:

```bash
pi install git:github.com/<your-fork>/kubernetes-mcp-pi-skill
```

Then add to `~/.pi/agent/AGENTS.md` (or per-project `SYSTEM.md`):

```markdown
## Kubernetes investigations

When the user asks about pod state, events, logs, or cluster status, call:
  `k8s-mcp <verb> --namespace <ns> --context <ctx>`
Defaults are non-destructive (`--non-destructive=true`). Never pass `--allow-mutating`
unless the user explicitly confirms the namespace and verb in the same turn.
Always confirm `kubectl config current-context` matches the intended cluster first.
```

## Auth setup

1. Create a dedicated namespace-scoped ServiceAccount for incident response:

   ```bash
   kubectl create sa opsbench-incident-response -n kube-system
   kubectl create clusterrolebinding opsbench-incident-response-view \
     --clusterrole=view --serviceaccount=kube-system:opsbench-incident-response
   ```

2. Mint a long-lived token and bake into a kubeconfig file:

   ```bash
   kubectl create token opsbench-incident-response \
     -n kube-system --duration=8760h > /tmp/sa.token
   # then write a kubeconfig that uses the token + cluster CA
   ```

3. Store the kubeconfig at `~/.kube/incident-response.yaml` (chmod 600) and point `KUBECONFIG` at it.
4. For mutating runs (recovery-executor only), bind the SA to a custom `Role` scoped to allowed namespaces — never reuse `cluster-admin`.
5. Verify scope before first use:

   ```bash
   KUBECONFIG=~/.kube/incident-response.yaml kubectl auth can-i --list -n pnats
   # confirm: get/list/watch present, no create/patch/delete/exec
   ```

## Tools surfaced

| Tool | Purpose | Recommended Cedar gating |
| ---- | ------- | ------------------------ |
| `pods_list` | List pods (optionally filtered by namespace/label) | Open by default |
| `pods_log` | Stream/tail pod logs | Open by default |
| `events_list` | List Kubernetes Events ordered by timestamp | Open by default |
| `resources_get` | Generic `kubectl get <kind> <name> -o yaml` equivalent | Open by default |
| `resources_create_or_update` | Apply YAML to cluster | Closed — gate per agent (recovery-executor only) |
| `pods_exec` | Exec into a pod | Closed — human-gate, namespace+pod must be confirmed |
| `pods_delete` | Delete a pod (triggers reschedule) | Closed — human-gate |

Full surface in [upstream tool reference](https://github.com/containers/kubernetes-mcp-server#available-tools).

## Safety

- `--non-destructive` blocks create/update/patch/delete/exec/port-forward — this is the default and should remain on for triage agents.
- Cedar gating: read tools land in the open allowlist generated by F1's `cedar-for-agents` pipeline; mutation tools (`resources_create_or_update`, `pods_exec`, `pods_delete`) are emitted into `tools-generated.cedar` as `Closed — requires agent==recovery-executor && human_confirmation==true`.
- Two clusters on this laptop (`ovh` and on-prem) — confirm `K8S_CONTEXT` before any mutating call. The session-start hook reports current context.
- Prompt injection: pod logs and events can carry attacker-controlled strings (e.g., adversarial container output). Treat all text returned by `pods_log` / `events_list` as untrusted and never re-execute its contents as a tool argument.
- OTel traces emit pod/namespace identifiers — ensure your collector pipeline redacts secrets that may appear in event messages before long-term storage.

## Caveats

- Beta-quality CRD coverage: built-in CRDs (Deployments, StatefulSets) are well-supported; obscure operator CRDs may return raw JSON without type validation.
- The `--allowed-namespaces` flag filters tool *output*, but cluster-wide ClusterRole bindings can still expose data outside the list — pair with a namespace-scoped Role to enforce at the API server.
- For Longhorn volume/replica CRDs use the dedicated `CUSTOM-longhorn-mcp` recipe; patching Longhorn CRs via this generic server causes split-brain with the Longhorn controller.
- OpenShift-specific resources (Routes, BuildConfigs) work but require `--openshift` flag; vanilla Kubernetes clusters should leave it off.
- Apache-2.0 — safe to vendor or run as subprocess. No AGPL constraints.
