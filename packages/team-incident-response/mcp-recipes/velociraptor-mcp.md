# MCP Recipe — velociraptor-mcp (DFIR — optional)

Velociraptor is a DFIR / endpoint-forensics platform. Used for deep host-level evidence
collection when standard collectors are insufficient (e.g., memory acquisition, file
timeline, registry-like artifacts on Linux).

## Source

- Project: <https://docs.velociraptor.app/>
- MCP wrapper: COMMUNITY / TO BE BUILT (no upstream MCP exists as of 2026-05-22)
- License: AGPL-3.0 (Velociraptor itself)

## Status

NOT YET DEPLOYED. Documented for future adoption when:

- A node-level corruption incident requires memory/disk imaging
- A security incident requires live triage with chain-of-custody
- Standard host collectors (journalctl, dmesg, iostat) prove insufficient

## Install (when adopted)

```bash
# Server
curl -L -o /usr/local/bin/velociraptor \
  https://github.com/Velocidex/velociraptor/releases/latest/download/velociraptor-linux-amd64
chmod +x /usr/local/bin/velociraptor
velociraptor config generate -i  # interactive
```

## Auth setup

- gRPC API certs (mTLS) — generated by `velociraptor config api_client`.
- Per-investigator user with role `analyst` (read VQL + collect artifacts, no admin).
- All collections audit-logged server-side.

## Read-only verification

VQL has read/write tiers; the MCP must restrict to `RunVQL` with `allow_write=false` and
expose only the artifact catalog (`ArtifactRepository`) — not the artifact editor.

## Caveats

- **Build-this-MCP** — no production MCP exists. If/when needed, the spec is:
  - Tools: `run_vql_query`, `collect_artifact`, `list_clients`, `get_flow_results`,
    `download_evidence_bundle`
  - All write-mode VQL (`copy`, `upload`, `execve`) must be Cedar-forbidden by default
- Memory acquisition is multi-GB — orchestrate via dedicated job, not MCP-streamed
- Chain-of-custody: Velociraptor signs collection bundles; preserve the signature when
  cataloging into custody.log
